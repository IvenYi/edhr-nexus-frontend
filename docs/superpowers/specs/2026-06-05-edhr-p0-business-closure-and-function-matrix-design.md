# eDHR 0-1 产品化功能矩阵与业务闭环设计规格

日期：2026-06-05  
状态：评审稿  
适用范围：商业级 ToB eDHR 新产品 0-1 模块划分、功能矩阵、P0 闭环设计与后续拆解依据

## 1. 结论摘要

冠骋的 eDHR 可以作为医疗器械 eDHR / 轻 MES 业务参考，但不能作为产品结构照搬。它有价值的不是低代码平台本身，而是把“制程配置 + 表单模板 + 事务流程 + 生产执行 + DHR 汇总 + 放行 + 追溯”串成了可落地的生产记录闭环。

我们的产品不应绑定单一行业，但首个商业化目标可以以医疗器械 eDHR 为主要验证场景。通用性不来自“无限低代码”，而来自以下可控配置能力：

- 基础主数据可扩展：工厂、车间、产线、产品、产品族、设备、SOP、工序、原因等。
- 模板可扩展：表单模板、DHR 模板、放行单模板、流转单模板。
- 制程可扩展：产品/产品族 + 生产模式 + 工艺路线 + DHR 模板 + 放行单 + 工序表单/SOP/权限。
- 事务可扩展：全局/按需事务、事务流程、事务节点、事务触发范围。
- 流程可扩展：流程模板、表单实例流程、模板/配置审批、事务流程、DHR 汇总审核、变更/作废审核、放行流程，并为检验、记录本、仓储事务保留扩展边界。

P0 不做“大而全系统”，而是先跑通这一条闭环：

```text
基础建模
-> 表单/DHR/放行模板
-> 工序/工艺路线/制程配置
-> 工单创建或同步
-> 工单批次拆分
-> 批次生产执行
-> 工序表单填报与表单审核
-> 工序事务触发与处理
-> DHR 自动聚合与人工补充整理
-> DHR 汇总审核
-> 放行单填报与放行审核
-> 归档冻结
-> DHR/表单/生产追溯
```

## 3. 产品边界

### 3.1 我们要做什么

我们要做的是可产品化、可定制、可私有化交付的 eDHR 系统。它的核心对象不是“任意低代码模型”，而是生产质量证据链：

```text
产品/制程配置
-> 批次/SN/工单
-> 工序执行记录
-> 事务记录
-> 检验/返工/异常/附录记录
-> DHR 目录与证据聚合
-> 审核与放行
-> 冻结归档
-> 追溯查询
```

### 3.2 我们不照搬什么

| 冠骋能力 | 我们的取舍 |
| --- | --- |
| Excel 式自由表单设计器 | 不照搬。P0 做结构化字段/表格/附件/签名块/审核块，P1 做 AI 辅助还原草稿。 |
| 大而全低代码模型平台 | 不照搬。只抽象 eDHR 需要的受控模板、表单实例、事务实例、DHR 证据。 |
| 全量 MES/WMS/QMS/LIMS/DMS | 不在 P0 做完整替代。只做 eDHR 所需引用、挂接、导入或轻量接口。 |
| 任意复杂 BPMN | 不在 P0 做完整 BPMN。先做可配置节点、条件、并行、审核、消息、事务嵌套。 |
| 设备互联/OCR/报表设计器全能力 | P2 扩展。P0 保留结构和接口边界。 |

## 4. 端到端业务闭环

```mermaid
flowchart LR
  A["基础数据"] --> B["模板中心"]
  A --> C["制程中心"]
  B --> C
  C --> D["工单/批次/SN"]
  D --> E["生产执行"]
  E --> F["表单填报/审核"]
  E --> G["事务触发/处理"]
  F --> H["DHR实例"]
  G --> H
  E --> H
  H --> I["DHR汇总"]
  I --> J["DHR审核"]
  J --> K["放行执行"]
  K --> L["归档冻结"]
  L --> M["追溯查询"]
```

业务流程图里的三段可落成如下：

| 阶段 | 业务含义 | 关键模块 |
| --- | --- | --- |
| 基础数据 | 产品、表单模板、DHR 模板、工序、工艺路线、事务先建好，再进入制程配置 | 主数据、模板中心、制程中心、事务中心 |
| 生产阶段 | 工单创建或 ERP 同步后拆分批次/SN，按制程开始生产，按工序填表，特殊情况触发事务 | 工单管理、批次/SN、生产执行、表单实例、事务实例 |
| 放行阶段 | 已完成 DHR 进入汇总，按模板整理证据，审核后生成/填写放行单，审核完成后归档 | DHR 汇总、DHR 审核、放行执行、归档、追溯 |

## 5. 功能矩阵

优先级定义：

- P0：首个商业可演示闭环必须具备；缺失会导致 DHR 从配置到放行无法闭环。
- P1：产品化增强；不影响首个闭环，但影响实施效率、客户适配和专业度。
- P2：高级扩展；面向复杂行业、深度集成、AI 增强或设备自动化。

| 功能 | P0/P1/P2 | 说明 |

## 10. 核心状态机

### 10.1 模板状态

```text
DRAFT -> IN_REVIEW -> EFFECTIVE -> OBSOLETE
IN_REVIEW -> RETURNED -> DRAFT
DRAFT -> VOIDED
```

规则：

- 已生效版本不可直接修改。
- 实例必须绑定模板版本。
- 新版本不自动改写历史实例。

### 10.2 表单实例状态

```text
OPEN -> SUBMITTED -> IN_REVIEW -> COMPLETED
IN_REVIEW -> RETURNED -> OPEN
COMPLETED -> CHANGE_PENDING -> COMPLETED
OPEN -> VOIDED
```

规则：

- 流程表单提交后按表单流程创建审核任务。
- 非流程表单可提交后直接完成。
- 完成后只能走受控变更。

### 10.3 事务实例状态

```text
CREATED -> RUNNING -> WAITING_TASK -> COMPLETED
RUNNING -> ENDED
WAITING_TASK -> RETURNED -> RUNNING
```

规则：

- 事务实例必须关联批次/SN、工序、来源事务版本。
- 事务处理过程产生的表单实例、返工任务、消息均作为 DHR 候选证据。
- 结束事务必须填写原因并留审计。

### 10.4 批次执行状态

```text
NOT_STARTED -> IN_PROGRESS -> COMPLETED
IN_PROGRESS -> ENDED
```

工序状态：

```text
WAITING -> IN_PROGRESS -> COMPLETED
IN_PROGRESS -> PARTIAL_COMPLETED
```

P0 可先不实现 `PARTIAL_COMPLETED`，但模型预留。

### 10.5 DHR 状态

```text
NOT_FILLED -> IN_PROGRESS -> COMPLETED -> SUMMARY_IN_PROGRESS -> SUMMARIZED
SUMMARY_IN_PROGRESS -> RETURNED -> COMPLETED
SUMMARIZED -> RELEASE_PENDING -> RELEASED -> ARCHIVED
RELEASED -> CHANGE_PENDING -> RELEASED
```

说明：

- `COMPLETED` 表示生产/DHR 填报完成，可以进入待汇总。
- `SUMMARY_IN_PROGRESS` 对应汇总审核中。
- `SUMMARIZED` 后才可进入放行。

### 10.6 放行状态

```text
NOT_RELEASED -> RELEASE_IN_PROGRESS -> RELEASED
RELEASE_IN_PROGRESS -> RETURNED -> RELEASE_IN_PROGRESS
RELEASED -> ARCHIVED
```

### 10.7 变更状态

```text
DRAFT -> SUBMITTED -> IN_REVIEW -> APPROVED -> APPLIED
IN_REVIEW -> RETURNED -> DRAFT
SUBMITTED -> CANCELLED
```

## 11. 系统流程引擎设计

### 11.1 调研结论

系统里的流程不能再理解为“4 个审批流”。从资料和在线页面看，冠骋至少把流程能力放在这些位置：

| 业务域 | 已观察或资料确认的入口 | 对我们的启发 |
| --- | --- | --- |
| 基础建模 | 流程建模 > 流程模板；页面含 DHR 审核模板、变更审核模板、作废审核模板，支持编辑、配置、建模追溯 | 需要独立的审核流程模板中心，流程定义必须版本化和可追溯。 |
| 模板建模 | 表单模板、DHR 模板、流转单模板、审批配置、模板审批 | 模板发布要走审批，发布后的模板版本不可被历史实例静默改写。 |
| 制程建模 | 工艺路线、制程配置 | P1 需要工艺/制程配置审批；P0 至少要保留发布与审计模型。 |
| 生产管理 | 我的事务、批次事务、SN 事务、生产配置 > 事务配置/事务列表 | 生产执行使用事务流程驱动过站时的表单填报、异常处置、返工/报废等动作。 |
| 检验管理 | 检验事务、检验规程、事务配置、事务列表 | 检验可先作为表单进入 DHR，P1 再成为独立检验事务流。 |
| 放行管理 | 放行列表、放行规程、事务配置、事务列表 | P0 用放行单表单流程闭环，P1 再做放行事务与放行规程。 |
| 记录管理 | 表单审核、DHR 审核、变更审核、流程干预、流程日志 | 审核任务、流转日志、转办/撤回/干预是受控记录系统的横向能力。 |
| 统计报表 | 消息记录、生产/表单/DHR/放行追溯 | 流程消息和流程事件需要进入可查询追溯。 |
| 集成中心 | 连接流、连接器、数据源、设备互联 | 第三方对接域，本轮不作为 eDHR 本体流程引擎设计重点，P2 保留边界。 |

在线页面进一步确认：流程模板配置页有“草稿/发布版本/当前操作”、保存、发布、关闭、开始/结束节点、右侧节点属性面板。这说明 P0 不能只做硬编码审批状态字段，而要有最小的流程定义、版本、生效和运行任务模型。

### 11.2 流程能力分层

建议把流程能力拆为“审核流程引擎”和“事务流程引擎”两条主干，再在业务域内绑定使用。

| 分层 | P0/P1/P2 | 说明 |
| --- | --- | --- |
| 流程定义中心 | P0 | 管理审核流程模板、流程版本、发布状态、节点配置、建模追溯入口。 |
| 审核流程引擎 | P0 | 负责表单审核、DHR 汇总审核、模板审批、变更/作废审核、放行审核。 |
| 表单实例流程 | P0 | 审核流程的专用场景，额外需要字段级权限、审核块、签名块。 |
| 事务流程引擎 | P0 | 负责生产执行中的事务实例、表单节点、条件分支、处理任务。 |
| 流程绑定规则 | P0 | 将流程定义绑定到模板、DHR 分类、具体模板、事务定义、制程配置、放行单等业务对象。 |
| 统一任务运行 | P0 | 生成待办、处理待办、推进节点、记录意见、签名、退回、转办。 |
| 流程日志 | P0 | 记录每个流程实例、节点、动作、处理人、时间、意见、签名、业务对象版本。 |
| 流程干预 | P1 | 管理员按权限转办、撤回到开始节点、关闭异常任务，必须填写原因并审计。 |
| 消息通知 | P1 | 流程到达节点、退回、超时、事务消息节点触发站内或外部通知。 |
| 集成连接流 | P2 | ERP、PLM、DMS、LIMS、设备等第三方系统对接，不进入 P0 核心。 |

### 11.3 审核流程引擎

审核流程引擎解决“受控对象由谁批准、如何退回、如何留痕”的问题。P0 不做完整 BPMN，但要支持可配置审批链。

P0 节点：

| 节点 | 用途 | P0/P1/P2 |
| --- | --- | --- |
| 开始 | 流程实例创建入口，记录发起人、发起时间、业务对象 | P0 |
| 审批 | 单人或多人候选审批，生成审批任务 | P0 |
| 并行审批 | 多个审批节点并行，全部通过后继续 | P0 |
| 结束 | 流程完成，回写业务对象状态 | P0 |
| 条件审批 | 按产品、风险等级、表单字段、变更类型选择路径 | P1 |
| 抄送/通知 | 不阻塞流程，只发送消息 | P1 |

P0 动作：

| 动作 | 说明 |
| --- | --- |
| 提交 | 发起流程，绑定流程定义版本并创建首个任务。 |
| 审核通过 | 当前节点完成，必要时电子签名。 |
| 退回 | 退回到发起人或指定前序节点，记录原因。 |
| 转办 | 转给其他有权限人员，记录转办人、接收人、原因。 |
| 撤回 | 未完成时由发起人或管理员撤回，P0 可先做受限撤回，P1 做流程干预台。 |
| 查看历史 | 查看节点路径、处理意见、签名、时间线和业务对象快照。 |

审核人配置：

- 指定用户。
- 角色或岗位。
- 部门或组织。
- 业务上下文负责人，如当前工厂质量负责人、产品族质量负责人、工序负责人。
- P1 支持动态规则，如按表单字段、产品风险等级、变更类型决定审核组织。

审核流程绑定场景：

| 场景 | P0/P1/P2 | 说明 |
| --- | --- | --- |
| 表单实例审核 | P0 | 表单提交后产生表单审核待办。 |
| DHR 汇总审核 | P0 | DHR 汇总提交时选择或命中汇总审核流程。 |
| 模板审批 | P0 | 表单模板、DHR 模板、放行单模板发布前审批。 |
| 表单变更审核 | P0 | 完成或归档后的表单更正必须走变更审核。 |
| DHR 变更审核 | P0 | 在 DHR 目录中变更表单或证据，审核通过后生效。 |
| 作废审核 | P0 | 表单或记录作废需要原因、签名、审批和状态冻结。 |
| 放行审核 | P0 | 放行单本质是受控表单，放行结论必须走审核/签名。 |
| 工艺/制程配置审核 | P1 | 路线、制程配置发布前审核，避免配置变更影响生产。 |
| 检验结果审核 | P1 | 独立检验作业上线后可复用审核流程。 |
| 记录本发布/变更审核 | P1 | 记录本作为补充记录进入 DHR 时需要受控审批。 |

### 11.4 表单实例流程

表单流程是审核流程的专用场景，它控制“表单由谁填、谁审、每个节点能看和改哪些字段”。不能把它等同于模板审批，也不能放进事务流程里硬编码。

P0 必须支持：

- 表单模板可选择基础表单、流程表单、视图表单；P0 重点支持基础表单和流程表单。
- 流程表单绑定一个审核流程定义版本。
- 每个节点可配置字段权限：编辑、只读、隐藏。
- 表单内显示流程审核块：当前节点、处理意见、审核结论、签名信息。
- 表单退回后只允许按节点权限修改，重新提交后保留前次退回意见。
- 表单完成后只读，后续修正走变更流程。

表单实例流程的典型路径：

```text
OPEN
-> 填报人提交
-> 生产主管审核
-> 质量主管并行审核
-> COMPLETED
```

事务里触发的表单节点、DHR 填报里的表单、生产工序表单、附录表单都应复用同一套表单实例流程能力，只是来源不同。

### 11.5 事务流程引擎

事务流程解决“生产、检验、放行、仓储等执行过程中，遇到某类业务动作时应触发哪些表单、条件、子任务和消息”的问题。它是 eDHR 流转核心之一，尤其用于过站时客户使用哪个事务填报哪个表单模板，最后进入 DHR 证据。

事务定义至少包含：

- 事务名称、编码、描述、状态。
- 作用范围：全局或按需。
- 适用场景：生产、返工、检验、放行、仓储。
- 适用产品家族、产品、工序、生产模式。
- 是否允许自定义流程覆盖默认流程。
- 流程草稿、发布版本、当前版本。
- 建模追溯和引用影响。

事务节点矩阵：

| 节点 | 用途 | P0/P1/P2 |
| --- | --- | --- |
| 开始/结束 | 事务实例生命周期入口和完成出口 | P0 |
| 表单节点 | 触发表单模板生成表单实例并填报 | P0 |
| 条件分支 | 按字段值、处理结论、产品/工序条件选择单一路径 | P0 |
| 事务节点 | 嵌套子事务，如不良处置触发返工事务 | P1 |
| 条件并行 | 多个条件满足时并行触发多个分支 | P1 |
| 配置节点 | 返工配置、批次转 SN、拆分、数量调整等 | P1 |
| 消息通知 | 触发站内、企微、钉钉、飞书、邮箱通知 | P1 |
| 检验执行节点 | 创建检验任务、绑定检验规程和检验表单 | P1 |
| 放行节点 | 创建或推进放行单/放行事务 | P1 |
| 物料/仓储节点 | 入库、出库、库存调整、物料平衡 | P2 |

P0 事务运行规则：

1. 事务实例必须关联来源：批次或 SN、工序、生产/返工场景、事务定义版本。
2. 生产执行页根据全局事务和按需事务配置展示可触发事务。
3. 触发后按发布版本创建事务实例，不受后续配置修改影响。
4. 表单节点生成的表单实例进入 DHR 候选证据，汇总时可自动或人工放入目录。
5. 条件分支从左到右或按优先级命中，未命中时必须有默认路径或阻断提示。
6. 事务可完成、退回、受控结束；结束必须填写原因并审计。
7. 我的事务、批次事务、SN 事务列表都只是事务运行的不同入口。

事务流先从生产 P0 做起。检验事务、放行事务和仓储事务在菜单上存在，但不应把 P0 扩成完整 MES/LIMS/WMS。它们应复用事务引擎模型，分阶段接入。

### 11.6 DHR 汇总与放行流程

DHR 汇总流程不是简单审批流，也不是纯拖拽目录。它由 DHR 模板、证据聚合、完整性检查和汇总审核共同组成。

P0 DHR 汇总路径：

```text
批次/SN执行完成
-> DHR状态变为已完成
-> 待汇总列表出现
-> 按DHR模板自动聚合必需表单
-> 从附录/事务/返工/记录本候选证据补充
-> 完整性检查
-> 直接汇总或提交DHR审核流程
-> 已汇总
```

DHR 审核复用审核流程引擎，但要有 DHR 专属规则：

- 审核对象是 DHR 汇总版本，而不是单张表单。
- 审核时查看目录、证据来源、表单状态、签名、变更状态、缺项提示。
- 审核退回后回到待汇总或已完成状态，允许重新整理汇总。
- 审核通过后冻结本次汇总版本，并触发放行前置状态。

放行 P0 采用放行单表单流程：

```text
DHR已汇总
-> 放行执行输入批次/SN
-> 系统按制程配置创建或找到放行单
-> 放行单填报
-> 放行审核/签名
-> 已放行
-> DHR归档冻结
```

P1 再补放行规程和放行事务，使放行可以像生产事务一样配置节点、消息、条件和补充表单。

### 11.7 变更与作废流程

变更和作废不是普通编辑功能，而是放行前后都可能发生的受控修正机制。

P0 变更类型：

| 类型 | 入口 | 生效方式 |
| --- | --- | --- |
| 表单变更 | 按表单流水号查询后发起 | 审核通过后将变更内容应用到表单当前版本，原值保留。 |
| DHR 变更 | 从 DHR 目录中选择表单或证据发起 | 审核通过后更新 DHR 证据目录或表单内容。 |
| 作废审核 | 对错误表单、无效记录、误创建记录发起 | 审核通过后状态作废，历史仍可追溯。 |

P1 增强：

- 记录本变更。
- 标红字段、点击查看前后差异。
- 放行后更正与重新放行影响评估。
- 多级质量风险审批。

变更流程必须记录：

- 变更原因。
- 变更人签名。
- 审核人签名。
- 变更前后值。
- 影响的 DHR、放行单、追溯索引。
- 审核意见、退回原因、转办记录。

### 11.8 检验、记录本、物料仓储流程边界

这些模块需要流程能力，但不应全部进入 P0。

| 模块 | P0 做法 | 后续做法 |
| --- | --- | --- |
| 检验管理 | 检验记录作为生产工序表单或 DHR 表单，进入表单流程和 DHR 汇总 | P1 做检验规程、检验事务、独立检验任务、检验审核。 |
| 记录本 | P0 仅保留 DHR 汇总候选证据接口 | P1 做记录本发布、填报、完成、变更、标签、审核。 |
| 物料消耗 | P0 用追溯字段记录关键物料批号、用量、设备 | P1 做物料消耗事务和物料平衡，P2 做完整仓储事务。 |
| 仓储作业 | P0 不做库存过账 | P2 做入库、出库、仓储事务和库存流水。 |
| 标签打印 | P0 记录标签相关字段和附件 | P1 做标签模板和打印任务，P2 接 BarTender 等外部系统。 |
| 第三方集成 | P0 手动导入和标准 API | P1 标准接口，P2 连接器/连接流和设备互联。 |

### 11.9 运行、日志、干预与消息

流程运行必须统一建模，不能每个业务模块各做一套待办和日志。

统一运行对象：

- 流程实例：绑定业务对象、流程定义版本、当前状态。
- 流程任务：节点任务、候选人、处理人、到达时间、截止时间。
- 节点动作：提交、通过、退回、转办、撤回、结束、系统自动流转。
- 流程日志：节点、动作、处理人、时间、意见、签名、IP/设备、业务对象快照。
- 干预记录：管理员、干预动作、原因、前后节点、影响对象。
- 消息记录：触发节点、接收人、渠道、内容模板、发送状态。

P0 至少要做到：

- 表单审核、DHR 审核、变更审核、作废审核、放行审核有可查日志。
- 事务实例处理过程有时间线。
- 高风险动作必须要求原因或签名。
- 历史日志不可被业务用户修改或删除。
- 追溯页面可以从 DHR、表单、事务反查流程日志。

P1 再提供完整流程干预台：

- 按流程类型、业务对象、当前节点、处理人查询。
- 转办到其他人员。
- 撤回到开始节点。
- 结束异常任务。
- 批量提醒或重新发送消息。

### 11.10 P0/P1/P2 流程交付边界

| 阶段 | 流程能力 |
| --- | --- |
| P0 | 审核流程定义/发布、表单实例流程、DHR 汇总审核、模板审批、变更/作废审核、放行单审核、生产事务表单节点和条件分支、运行日志、最小转办。 |
| P1 | 工艺/制程审批、记录本流程、检验事务、放行事务、事务节点/条件并行/配置节点/消息节点、流程干预台、建模追溯 UI、流程超时提醒。 |
| P2 | 仓储事务、连接流、设备互联流程、复杂条件规则、跨系统补偿重试、自定义报表流程分析。 |

## 12. 关键业务规则

### 12.1 制程配置规则

制程配置是 P0 的核心，不只是“工艺路线绑定表单”。

制程版本至少要定义：

- 适用范围：产品家族或产品。
- 产品版本和生产模式：批次、SN、批次转 SN 等。
- 工艺路线版本。
- DHR 模板版本。
- 放行单模板版本。
- 每个工序的 SOP。
- 每个工序的待填表单、必填标识、排序。
- 每个工序表单的填报权限、字段编辑权限。
- 每个工序可见事务：来自全局事务、按需事务、例外剔除规则。

### 12.2 表单流程规则

表单流程不是模板审批流程，也不是事务流程。它控制“表单实例填完后由谁审、审什么字段”。

P0 支持：

- 开始、审批、并行、结束。
- 审核人配置：用户、角色、部门、组织。
- 节点字段权限：编辑、只读、隐藏。
- 审核、退回、转办。
- 流程日志和审计追踪。

P0 不做：

- 任意脚本节点。
- 无限复杂 BPMN 网关。

### 12.3 事务流程规则

事务是生产执行过站时处理特殊动作的核心引擎。

P0 支持：

- 表单节点。
- 条件分支。
- 事务实例处理。
- 全局/按需作用范围。
- 生产/返工场景区分。
- 产品家族、产品、工序条件。

P1 支持：

- 条件并行。
- 子事务节点。
- 配置节点：返工配置、拆分配置。
- 消息通知节点。

### 12.4 DHR 汇总规则

DHR 汇总不是从空目录完全手工拖拽。正确理解应是：

1. DHR 模板定义目录结构和必需表单。
2. 制程执行过程中生成表单实例、事务表单、返工表单、附录表单、记录本表单。
3. 当批次/SN 执行完成且 DHR 状态为已完成，进入待汇总。
4. 汇总页面按 DHR 模板自动聚合必需表单。
5. 用户可从附录、事务、返工、记录本候选证据中补充拖入目录，也可删除不应进入归档目录的补充项。
6. 提交时可选择汇总审核流程，也可直接汇总。
7. 审核通过后 DHR 状态变为已汇总，进入放行前置状态。

P0 完整性检查至少包括：

- 必需目录是否存在。
- 必需表单是否完成。
- 表单流程是否完成。
- 事务流程是否完成或已受控结束。
- 必填字段是否通过校验。
- 签名/审核是否完成。
- 变更是否已关闭。

### 12.5 放行规则

放行以放行单为 P0 主入口：

1. DHR 已汇总后才允许放行。
2. 放行执行输入批次/SN。
3. 系统检查是否已有放行单。
4. 没有则按制程配置中的放行单模板自动创建。
5. 有则直接进入填报或查看。
6. 放行单提交后走审核/签名。
7. 放行完成后冻结 DHR 证据，后续更正必须走变更。

P1 再支持：

- 放行事务。
- SN 合并放行。
- 条件放行、紧急放行。

### 12.6 追溯规则

追溯不是简单报表，它依赖表单中的追溯字段和生产执行事件。

P0 追溯索引来源：

- 批次/SN。
- 产品/产品版本。
- 工单号。
- 工序。
- 表单流水号。
- 设备。
- 记录单号。
- 追溯日期。
- 事务实例。
- DHR 实例。
- 放行单。

P0 查询：

- DHR 正向/反向追溯。
- 表单正向/反向追溯。
- 批次生产时间线。

## 13. 表单设计器策略

P0 不做 Excel 式设计器，原因是：

- 难以保证字段语义、流程权限、审计追踪和结构化追溯。
- 实施容易变成“画表格外包”，产品复用差。
- AI 还原也需要结构化落点，而不是像素级复刻。

P0 做结构化模板设计：

| 组件 | 用途 |
| --- | --- |
| 章节 | 表单分区、标题、说明 |
| 字段 | 文本、数字、日期、枚举、人员、部门、产品、设备、批次/SN |
| 表格 | 固定表、动态表、主子表 |
| 附件/图片 | 报告、照片、证明文件 |
| 条码/扫码字段 | 批次、SN、设备、物料、工单 |
| 签名块 | 操作签名、复核签名、质量签名 |
| 审核块 | 流程节点意见、结论、退回原因 |
| 校验规则 | 必填、格式、范围、公式、主子表联动 |
| 追溯字段 | 建立正反向追溯索引 |

P1 做 AI 辅助还原：

```text
上传 Word/Excel/PDF/图片
-> AI 识别表头、字段、表格、签名、审核区域
-> 生成结构化模板草稿
-> 人工调整字段/规则/流程
-> 模板审批
-> 发布受控版本
```

AI 不允许：

- 直接发布模板。
- 静默修改已生效模板。
- 修改正式表单实例。
- 做最终放行判断。

## 14. 前端模块建议

P0 前端模块按业务闭环组织：

```text
dashboard        工作台、待办、我的事务、表单审核
workflow-center 流程模板、流程版本、流程绑定、流程任务、流程日志、流程干预
master-data      工厂、车间、产线、产品、设备、SOP、工序、原因
template-center  表单模板、DHR模板、放行单模板、表单流程、模板审批
process-center   工艺路线、制程配置
txn-center       事务列表、事务流程配置、事务配置、事务实例
production       工单、批次、SN、批次执行、返工执行、批次事务
record-center    表单填报、表单审核、表单列表、DHR填报、DHR列表
dhr-summary      DHR汇总、DHR审核、DHR详情
release          放行执行、放行列表
change-control   表单变更、DHR变更、变更审核、变更列表
traceability     DHR追溯、表单追溯、生产追溯
system           用户、角色、权限、审计、编码规则、导入导出
```

交互原则：

- ToB 生产质量系统，以表格、筛选、详情、抽屉、步骤条、状态标签为主。
- 不做营销式首页。
- 高风险动作要二次确认、原因、签名、审计。
- 生产执行页要适合扫码、少输入、快速定位当前工序/待填表单/事务。
- DHR 汇总页要突出目录树、证据候选区、完整性检查、审核提交。

## 15. 阶段拆解建议

### Phase 0：可信底座与基础对象

目标：先让受控记录和状态机成立。

范围：

- 用户/角色/权限。
- 审计追踪。
- 电子签名。
- 流程模板、流程任务、流程日志。
- 文件附件。
- 基础主数据。
- 表单模板版本。
- 表单实例。
- DHR 模板和 DHR 实例。

### Phase 1：批次生产闭环

目标：跑通一条批次 eDHR。

范围：

- 工艺路线。
- 制程配置。
- 工单/批次拆分。
- 批次生产执行。
- 工序表单填报。
- 表单流程审核。
- DHR 汇总和审核。
- 放行单填报和放行。
- 变更/作废审核。
- DHR/表单追溯。

### Phase 2：事务驱动异常闭环

目标：让生产特殊情况可配置、可处理、可进入 DHR。

范围：

- 事务定义。
- 事务流程表单节点和条件分支。
- 全局/按需事务配置。
