## 2. 产品摘要

### 2.1 一句话定义

建设一款不绑定单一行业的商业级可定制 eDHR 系统，用于管理批次、台套或序列号级别产品从基础建模、模板、制程、事务、生产执行、DHR 汇总、审核放行到归档追溯的完整证据链。

### 2.2 要解决的问题

大量制造企业仍依赖纸质记录、Word/Excel、共享文件夹、ERP/MES/QMS/LIMS 零散数据来证明每一批、每一台或每一个序列号产品已经按照受控要求完成生产、检验、审核和放行。这会带来几个典型问题：

- 批生产记录填写慢、漏填多、复核成本高。
- QA 放行前需要人工跨纸质记录、检验报告、标签、偏差记录、ERP 数据逐项核对。
- 电子记录的数据完整性能力不足，修改原因、原值/新值、操作人、日期、审计追踪和电子签名证据不完整。
- 产品、批号、序列号、UDI、物料批号、关键部件、设备、工艺参数、检验结果、异常处理和放行结论之间的追溯链条断裂。
- 高监管行业更明确要求电子记录/数据完整性、软件确认/验证、批次/台套/SN 生产记录、检验记录、产品放行、追溯、异常/不合格/返工和跨组织证据衔接。

### 2.3 产品方案

产品定位为专注 eDHR 的生产记录与放行证据系统，不做完整 MES，不做完整 QMS，也不做通用低代码平台。

产品由九类能力组成：

- 流程与可信基座：权限、审计追踪、电子签名、审核流程、表单实例流程、事务流程、受控记录、数据完整性、验证/确认支持包。
- 基础建模：工厂、车间、产线、产品、产品版本、设备、SOP、工序、原因。
- 模板中心：结构化表单模板、表单流程、DHR 模板、放行单模板、模板版本。
- 制程中心：工艺路线、制程配置、工序表单/SOP/权限、事务显隐规则。
- 事务中心：事务定义、全局/按需事务、事务流程、事务实例。
- 生产执行：工单、批次、SN 基础模型和查询、工序过站、表单填报、事务触发、附录。
- DHR 与记录管理：DHR 自动生成、DHR 填报、DHR 汇总、DHR 审核、表单列表。
- 放行、变更与追溯：放行单生成、放行审核、变更/作废审核、归档冻结、DHR/表单/生产追溯。
- 集成与 AI 扩展：导入、OpenAPI、受控引用；AI 表单还原放 P1，不阻塞 MVP。

### 2.4 成功指标

| 指标 | 当前基线 | MVP 目标 | 测量方式 |
| --- | --- | --- | --- |
| 单批/单台 DHR 审核时间 | 试点前采样 | 试点场景降低 30% | 试点前后 QA 审核时间对比 |
| 放行前缺项识别 | 依赖人工 | 已配置必填项 100% 被系统识别 | 完整性检查日志 |
| 受控记录审计追踪覆盖率 | 通常不完整 | 受控新增、修改、删除、状态流转、签名、放行 100% 留痕 | 测试用例与运行抽样 |
| 电子签名证据完整性 | 通常不一致 | 签名人、时间、含义、对象、认证事件 100% 记录 | 签名记录检查 |
| DHR 归档包准备时间 | 人工整理 | 放行后 5 分钟内生成试点 DHR 包 | 产品日志与用户观察 |

## 3. 范围与边界

### 3.1 产品定位

产品管理的是“某一批、某一台或某一序列号产品实际如何按已发布制程执行、填报表单、触发事务、汇总 DHR、审核放行和归档追溯的证据链”。

推荐对外描述：

> 面向高监管制造和复杂质量记录场景的商业级 eDHR 系统，支持结构化表单、DHR 模板、制程配置、事务驱动生产执行、DHR 自动汇总、审核放行、审计追踪、电子签名、归档导出和追溯查询。

### 3.2 范围内

- 工厂、车间、产线、产品、设备、SOP、工序、原因等基础建模。
- 结构化表单模板、DHR 模板、放行单模板及受控版本。
- 表单流程：配置填报字段、审核字段、审核组织/人员和字段权限。
- 工艺路线和制程配置：产品/产品族、生产模式、路线、DHR 模板、放行单、工序表单/SOP/权限、事务显隐规则。
- 事务定义和事务配置：全局/按需事务、表单节点、条件分支、事务实例。
- P0 按工单或生产批次生成 DHR 实例；SN 保留数据模型和基础查询，不进入首个执行闭环。
- 生产过程记录填写，包括工序过站、扫码、附件、图片、签名、复核和退回。
- 原材料、关键部件、中间产品、成品、批号、序列号和 UDI 相关追溯。
- 工艺参数、设备、工装、环境、清场、包装、标签确认等可配置记录。
- 进货检验、过程检验、成品检验在 P0 以表单方式进入 DHR 证据链，P1 再做独立检验事务。
- 偏差、不合格、返工、返修、让步、紧急放行、CAPA 编号挂接；完整 QMS 流程不在 MVP 内。
- Review by Exception：缺项、未复核、未签名、超限、异常未关闭、模板/版本不一致、放行阻断项。
- DHR 汇总：按 DHR 模板自动聚合必需证据，允许从附录/事务/返工/记录本候选证据补充。
- 产品放行流程和放行证据固化。
- 表单/DHR 变更和作废审核。
- DHR 归档包导出。
- 合规基座：RBAC、审计追踪、电子签名、记录保存、备份/恢复要求、流程引擎、验证/确认支持包。

### 3.3 范围外

- 不做完整 MES：高级排产、派工、OEE、产能计划、复杂线边仓、现场绩效管理不在 MVP 内。
- 不做完整 QMS：CAPA、投诉、供应商质量、内审、管理评审、文控、培训、变更控制的完整生命周期不在 MVP 内。
- 不做完整 WMS/ERP：库存金额、采购、收货、货位优化、成本核算、库存事务不在 MVP 内。
- 不做完整 LIMS：样品生命周期、试剂/培养基管理、仪器管理、OOS/OOT 调查、实验室排程不在 MVP 内。
- 不做完整 UDI 平台：监管数据库上传、全量标签设计不在 MVP 内；MVP 可记录和核对 UDI/PI。
- 不做完整 DMS：文件起草、审批、分发、培训联动、作废副本控制不在 MVP 内；MVP 可引用受控文件编号和版本。
- 不做 Excel 式自由表单设计器。MVP 做结构化字段、表格、附件、签名块、审核块、校验规则；AI 表单还原放 P1。
- 不做完整 BPMN/iPaaS/集成中心。MVP 先做审核流程、表单流程、事务流程的必要节点和运行日志。
- 不自动判定企业是否符合 GMP。
- 不替代客户质量负责人或授权放行人员的最终判断。

### 3.4 合规宣称边界

允许的产品宣称：

- 支持结构化生产记录、检验记录和 DHR 汇总管理。
- 支持电子记录的数据完整性控制、审计追踪、电子签名、流程日志、追溯和放行证据。
- 帮助质量团队准备 DHR 证据包并进行例外审核。
- 提供供应商侧验证/确认支持材料；在医疗器械场景下可映射 GMP、CSV/CSA。

需要避免的过度宣称：

- 客户购买或使用系统后，客户企业整体质量体系或法规合规状态自动成立。
- 系统自动判断产品是否可以放行。
- 系统替代质量部门审核或授权放行人员判断。
- 替代 QMS/MES/ERP/LIMS/DMS/WMS/iPaaS。

需要明确的是：软件自身必须在已声明的 eDHR 预期用途范围内进行受控设计，并提供数据完整性、权限、审计追踪、电子签名、记录保存、流程日志、追溯、放行证据和验证支持能力。这里限制的是对“客户整体质量体系/法规合规结果”的过度承诺，不是否定软件自身应满足适用行业要求。

## 4. 目标用户与购买角色

| 角色 | 任务 | 痛点 | 成功信号 | 购买影响 |
| --- | --- | --- | --- | --- |
| 生产操作员 | 按要求完成生产记录 | 纸质记录影响操作、重复填写、怕漏填错填 | 录入快、可扫码、必填清楚 | 中 |
| 生产主管 | 跟踪 DHR 进度并纠正缺口 | QA 审核前难以及时发现记录不完整 | 实时看到 DHR 状态和待办 | 中 |
| QA 审核员 | 审核批次/台套证据并支持放行 | 逐页人工检查慢且容易漏 | 例外审核、缺项清楚、证据完整 | 高 |
| 放行审核人 | 确认放行条件并签发放行 | 需要确信生产、检验、异常、标签、UDI 均已闭环 | 放行清单和证据可追溯 | 高 |
| 质量负责人 | 保持记录合规和检查准备状态 | 纸电混合证据脆弱 | 审计追踪、签名、归档、检索可靠 | 高 |
| IT/验证负责人 | 部署并确认受控质量相关系统 | 供应商验证资料不完整 | 系统说明、权限矩阵、测试脚本、发布记录齐全 | 高 |
| 工艺/工业工程师 | 将工艺要求转成受控执行记录 | 模板漂移、字段规则难维护 | 模板版本和规则可控 | 中 |
| 实施/配置顾问 | 建模、模板、制程和事务配置 | 每个项目都写定制代码 | 复用模板、事务和制程配置 | 中 |
| 企业负责人 | 降低合规风险和质量成本 | 重型系统贵且慢 | 快速试点、审核效率提升、范围受控 | 高 |

初始目标客户：

- 有批次、台套、SN、工单、物料、设备、检验、放行证据链要求的制造企业。
- 生产记录纸质化严重、DHR 汇总和放行审核耗时长、追溯压力较大的企业。
- 面临客户审计、行业监管、委外协同或出口客户质量审计压力的企业。
- 暂时不想上重型 MES，也不想重做 QMS/WMS/LIMS/DMS，但需要快速电子化 DHR 的企业。
- 医疗器械可作为首个高监管验证场景，但平台设计不绑定医疗器械。

## 5. 市场与竞品环境

| 替代方案 | 用户用途 | 优势 | 缺口 | 我们的差异化 |
| --- | --- | --- | --- | --- |
| 纸质记录 + Word/Excel + 共享文件夹 | 当前 DHR 记录和审核 | 熟悉、便宜、灵活 | 审计追踪弱、审核慢、追溯差 | 受控电子记录链和例外审核 |
| 完整 MES | 生产执行和现场控制 | 能力强、集成深 | 贵、周期长、范围大 | 专注 eDHR，实施更轻 |
| 完整 QMS | CAPA、偏差、变更等质量流程 | 质量流程治理强 | 不擅长生产记录执行 | 只挂接 QMS 事件，不吞 QMS |
| ERP/WMS | 工单、BOM、库存、批号 | 主数据权威 | 不适合 DHR 证据链 | 接入业务数据，固化 DHR 证据 |
| LIMS | 样品和检验结果 | 实验室控制强 | 不覆盖生产放行证据 | 将检验证据纳入 DHR |
| 国内 eDHR/eBR 厂商如冠骋 | 电子批记录和快速车间数字化 | 已验证国内需求，事务配置 + 制程配置 + DHR 汇总有参考价值 | 可能向轻 MES 或低代码平台扩展 | 更强调可产品化配置、结构化表单、流程日志和边界清晰 |
| 国际 eDHR 厂商 | eDHR、电子记录、质量/制造执行 | 成熟度高 | 成本高、本地交付和私有化适配不一定轻 | 本地交付、私有化友好、可按行业快速配置 |

待验证假设：

- 国内早期客户更可能接受私有化部署或专有云，而不是公有 SaaS 存放真实生产质量记录。
- 如果 ERP/MES/QMS 集成边界清晰，客户会接受聚焦 eDHR 的独立产品。
- 验证/确认支持包会显著提升销售和试点转化。

| 追溯查询 | DHR、表单、事务、生产时间线正反向追溯 | P0 |
| 导入与 OpenAPI | P0 只做工单/主数据导入和 API 契约，深度集成后置 | P1 |
| 检验事务/记录本/物料事务 | P0 用表单承载，P1 做独立模块 | P1 |
| AI 表单还原 | AI 生成结构化模板草稿，人工确认后发布 | P1 |

## 7. MVP 策略与第一步开发范围

### 7.1 MVP 主线

第一版可销售 MVP 应跑通这一条主线：

```text
基础建模
-> 表单/DHR/放行单模板
-> 工艺路线与制程配置
-> 事务定义与事务配置
-> 工单创建或导入
-> 批次生产执行
-> 工序表单填报与审核
-> 工序事务触发与处理
-> DHR 自动汇总与人工补充整理
-> DHR 汇总审核
-> 放行单填报与放行审核
-> 归档冻结
-> DHR/表单/生产追溯
```

### 7.2 第一阶段优先开发范围

第一步建议先开发“流程与可信基座 + 批次生产闭环内核”。这部分既不绑定具体行业，又能支撑后续客户定制。

Phase 0 范围：

- 核心领域对象和生命周期状态。
- 组织、用户、角色、权限模型。
- 审计追踪服务。
- 电子签名服务。
- 审核流程、表单流程、事务流程的最小引擎。
- 基础建模：工厂/车间/产线、产品、设备、SOP、工序。
- 结构化表单模板、DHR 模板、放行单模板。
- 工艺路线和制程配置。
- 事务定义、事务流程配置和事务实例。
- 工单、批次、工序执行和表单填报。
- DHR 实例、DHR 汇总、完整性检查、放行单、归档。
- DHR、表单、事务、生产时间线追溯。
- 架构与 API 边界文档。

暂时不要先做：

- 完整 SN 生产深度。
- 连续生产和复杂并行工序。
- 完整检验管理、检验事务和检验规程。
- 完整记录本。
- 完整返工/返修路线。
- 复杂物料平衡和 WMS 库存事务。
- ERP/MES/WMS/QMS/DMS/LIMS 深度集成。
- 设备/仪器联网。
- AI 表单还原和 AI 证据检查。
- 大屏和复杂看板。

### 7.3 MVP 非目标

MVP 不能变成通用低代码表单平台，也不能变成轻 MES。每个功能都必须能映射到 DHR 证据、生产执行、表单审核、事务处理、DHR 汇总、放行审核、追溯或受控电子记录数据完整性。

## 8. 核心场景

### 场景 1：完成基础建模

- 触发：实施团队准备上线一个产品族或产线。
- 角色：系统管理员、工艺工程师、实施顾问。
- 当前方式：主数据散落在 ERP、Excel、SOP 和现场经验中。
- 目标流程：配置工厂、车间、产线、产品、产品版本、设备、SOP、工序、原因字典。
- 输出证据：主数据版本、启停状态、审计追踪。
- 验收标准：
  - 制程、表单、事务和追溯都能引用这些基础对象。
  - 基础对象停用后不影响历史 DHR 可读性。

### 场景 2：配置表单模板、表单流程和 DHR/放行模板

- 触发：工艺/质量团队需要电子化某个生产、检验、放行或附录记录。
- 角色：工艺工程师、QA 审核员、系统管理员。
- 当前方式：Word/Excel 表单在文件系统或 DMS 受控后打印或手工填写。
- 目标流程：创建结构化模板草稿，配置字段/表格/附件/扫码/签名块/审核块/校验规则，配置表单流程，审批后发布。
- 输出证据：模板版本、流程版本、批准/签名记录、审计追踪、生效状态。
- 异常路径：模板退回、版本作废、生效后变更。
- 验收标准：
  - 已生效模板不能被静默编辑。
  - 表单流程能控制填报和审核节点的字段权限。
  - DHR 模板定义目录、必需证据和自动聚合规则。
  - 放行单模板可在 DHR 汇总通过后自动生成实例。

### 场景 3：配置制程和事务

- 触发：产品或产品族需要配置生产执行路径。
- 角色：工艺工程师、生产主管、QA、实施顾问。
- 当前方式：路线、SOP、记录表和异常处理动作靠人工理解。
- 目标流程：创建工艺路线，配置制程版本，绑定 DHR 模板、放行单模板、工序表单、SOP、权限和事务显隐规则；配置全局/按需事务。
- 输出证据：制程版本、事务版本、发布记录、引用关系。
- 异常路径：制程退回、事务停用、已发布配置变更。
- 验收标准：
  - 批次执行必须绑定已发布制程版本。
  - 生产执行页能按当前工序显示待填表单和可触发事务。
  - 事务实例保存来源事务版本和触发上下文。

### 场景 4：批次生产执行和表单填报

- 触发：工单、批次、台套或序列号进入生产。
- 角色：生产主管、操作员、复核员、QA。
- 当前方式：人工组装纸质 DHR 包或 Excel 记录。
- 目标流程：按制程生成批次执行和 DHR 实例，工序过站，填写工序表单，采集物料、参数、设备、人员、附件和签名。
- 输出证据：批次执行、工序实例、表单实例、字段值、附件、签名、审计事件。
- 异常路径：缺字段、物料批号错误、超限值、退回复填、受控补录。
- 验收标准：
  - 必填字段未完成时不能完成记录或过站。
  - 超限值产生异常标记或阻断项。
  - 签名后修改必须进入受控更正流程并产生审计追踪。

### 场景 5：生产过程中触发事务

- 触发：生产执行中发生报工、不良、报废、返工、补充记录等特殊动作。
- 角色：操作员、生产主管、QA。
- 当前方式：线下开单、口头通知、纸质补充记录。
- 目标流程：生产执行页触发事务，事务流程按节点要求填报表单、条件分支、处理任务和受控结束；事务过程形成 DHR 候选证据。
- 输出证据：事务实例、事务表单、处理任务、流程日志、审计追踪。
- 异常路径：事务退回、事务受控结束、事务未完成阻断工序或放行。
- 验收标准：
  - 事务实例必须关联批次、工序、来源事务版本，并预留 SN 上下文字段。
  - 我的事务、批次事务列表能查询同一事务实例；SN 事务列表作为 P1 执行能力。
  - 未完成或未受控结束的阻断事务会进入完整性检查。

### 场景 6：DHR 汇总和审核

- 触发：批次生产完成或进入 DHR 审核阶段。
- 角色：QA 审核员、生产主管、归档负责人。
- 当前方式：人工从多张表单、附件和异常记录中整理 DHR。
- 目标流程：DHR 模板按目录自动聚合生产表单、事务表单、返工表单、附录表单；用户从候选证据补充整理；系统执行完整性检查；提交 DHR 汇总审核。
- 输出证据：DHR 汇总版本、目录项、完整性检查结果、审核意见、签名和流程日志。
- 异常路径：缺少必需证据、表单未审核、事务未完成、退回补充。
- 验收标准：
  - DHR 汇总不是空目录手工拖拽，而是模板驱动自动聚合。
  - 人工补充证据必须保留来源、原因和审计。
  - DHR 审核对象是 DHR 汇总版本。

### 场景 7：放行、变更和归档追溯

- 触发：DHR 汇总审核通过，准备进入放行决定。
- 角色：放行审核人、质量负责人、归档负责人。
- 当前方式：人工整理放行决定和 DHR 证据包。
- 目标流程：系统生成放行单，放行填报和审核，签名后冻结 DHR，生成归档包；放行后如需更正必须走变更/作废流程。
- 输出证据：放行单、放行签名、只读 DHR 状态、DHR 归档包、变更/作废记录、追溯时间线。
- 异常路径：放行退回、放行后更正/补充、客户审计请求。
- 验收标准：
  - 已放行 DHR 除受控更正/补充流程外不可修改。
  - 归档包包含 DHR 目录、表单、事务证据、放行单、签名、附件、审计追踪、流程日志和元数据。
  - 可从 DHR 反查表单、事务、批次工序、签名、审计和流程日志。

## 9. 用户故事

1. 作为系统管理员，我希望配置组织、工厂、用户、角色、权限和数据范围，以便只有授权用户可以创建、修改、复核、签名、放行、导出或管理受控记录。
2. 作为实施/工艺工程师，我希望配置工厂、车间、产线、产品、设备、SOP、工序和原因，以便后续模板、制程、事务和追溯都能复用基础建模。
3. 作为工艺工程师，我希望创建结构化表单模板和 DHR 模板，以便每个表单实例和 DHR 汇总都基于已批准的模板版本。
4. 作为 QA 审核员，我希望表单模板、DHR 模板、放行单模板、流程和制程配置的审批/变更都有审计追踪，以便证明配置受控。
5. 作为流程管理员，我希望配置审核流程、表单实例流程和事务流程，以便不同对象可以按业务规则流转。
6. 作为工艺工程师，我希望配置工艺路线和制程，以便产品、路线、DHR 模板、放行单、工序表单、SOP、权限和事务显隐能统一发布。
7. 作为生产主管，我希望按工单生成批次执行和 DHR 实例，以便围绕产品证据链组织执行记录。
8. 作为操作员，我希望生产执行页显示当前工序、待填表单、SOP 和可触发事务，以便生产时正确完成记录。
9. 作为操作员，我希望扫码录入物料批号、设备或序列号，以便减少重复手填并形成追溯。
10. 作为生产/QA 用户，我希望在生产过程中触发报工、不良、返工、补充记录等事务，以便特殊业务动作也能进入 DHR 证据链。
11. 作为复核/审核人员，我希望使用带签名含义的电子签名，以便复核、批准和放行证据清晰。
12. 作为 QA 审核员，我希望 DHR 按模板自动汇总，并按例外审核缺失、异常、未签名、事务未完成或未关闭事项。
13. 作为放行审核人，我希望 DHR 汇总通过后自动生成放行单，以便基于配置的放行证据作出并签署放行决定。
14. 作为质量负责人，我希望表单/DHR 变更和作废都经过原因、前后对比、审核、签名和日志，以便受控更正。
15. 作为验证/IT 负责人，我希望获得供应商验证/确认支持资料，以便完成客户侧系统确认/验证。
16. 作为客户审计或检查准备人员，我希望导出完整 DHR 归档包，以便快速提供证据。
17. 作为业务模块设计伙伴，我希望有明确 API 和模块边界，以便设计检验、记录本、物料、返工、集成模块时不用重复实现审计、签名和流程运行。

## 12. AI 需求

AI 不在 MVP 范围内。

如果未来加入 AI/OCR/文档理解能力，AI 不得静默修改受控记录或最终放行决定。所有 AI 建议必须有来源引用、置信度或不确定性提示、人工确认、模型/提示词版本追踪，并且除非合同明确允许，不得用于模型训练。

### 13.2 状态机

模板：

```text
草稿 -> 审核中 -> 已生效 -> 已作废
草稿 -> 已作废
审核中 -> 已退回 -> 草稿
```

事务实例：

```text
已创建 -> 运行中 -> 待处理 -> 已完成
运行中/待处理 -> 已退回 -> 运行中
运行中/待处理 -> 已受控结束
```

DHR 实例：

```text
已创建 -> 执行中 -> 汇总中 -> 审核中 -> 待放行 -> 已放行 -> 已归档
审核中 -> 已退回 -> 汇总中
待放行 -> 已退回 -> 审核中
已放行 -> 更正待处理 -> 更正已批准 -> 已放行
```

表单实例：

```text
打开 -> 已提交 -> 已复核
已提交 -> 已退回 -> 打开
已复核 -> 更正待处理 -> 已更正 -> 已复核
打开/已提交 -> 已作废
```

异常链接：

```text
打开 -> 处理中 -> 已关闭
打开/处理中 -> 放行豁免
```

放行决定：

```text
待处理 -> 已批准
待处理 -> 已拒绝
待处理 -> 条件批准
已批准/条件批准 -> 已归档
```

## 15. 部署与数据安全

| 模式 | 目标客户 | 数据处理 | 优势 | 风险 | MVP 建议 |
| --- | --- | --- | --- | --- | --- |
| 私有化部署 | 对生产质量数据敏感的高监管制造客户 | 客户控制基础设施和数据库 | 信任度最高，符合保守客户预期 | 交付和支持更重 | MVP 主模式 |
| 专有云/单租户云 | 接受托管服务的中型客户 | 每客户独立数据库/存储 | 更新和支持更快 | 需要安全评审 | 设计兼容，非第一依赖 |
| 公有 SaaS | 演示、沙箱、低风险试用 | 不建议存真实生产记录，除非合同明确 | 低摩擦 | 对真实 DHR 信任低 | 仅作演示 |

安全基线：

- 所有受控操作服务端鉴权。
- 密码和会话策略可配置。
- 签名、放行等敏感操作支持二次认证。
- 登录、失败登录、权限变更、导出、接口导入、管理员操作均留日志。
- 传输加密；静态加密根据部署环境支持并写入部署指南。
- 提供备份/恢复流程和恢复测试证据模板。

## 17. 商业模式与包装

首个可售卖包装建议：

- 私有化部署或专有云选项。
- 核心 eDHR 包：基础建模、结构化表单、DHR 模板、制程配置、事务配置、批次生产、DHR 汇总、放行、归档导出。
- 可信基座：RBAC、审计追踪、电子签名、受控存储、审核流程、表单流程、事务流程、验证支持包。
- 实施包：一个试点产品族/产线、3-5 个表单模板、一个 DHR 模板、一个放行单模板、一个制程配置、1-2 个事务模板、一个归档包格式。
- 年度维护和验证支持续费。

避免纯项目制。实施工作应限制在模板、主数据、角色、工作流配置和少量接口范围内。

## 18. 度量与分析

产品指标：

- DHR 实例创建数。
- DHR 记录完成数。
- Review by Exception 发现项。
- 放行周期。
- 归档包导出数。
- 签名后受控更正数。
- 每个 DHR 的审计事件数。

运营指标：

- 平均 DHR 执行完成时间。
- 平均 QA 审核时间。
- 平均放行审批时间。
- 退回记录数量。
- 放行审核时仍打开的异常数量。

商业指标：

- 试点转付费率。
- 单试点实施天数。
- 不同客户规模的年度合同金额。
- 可复用行业模板包数量。

## 19. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 范围蔓延成 MES/QMS/WMS/LIMS | 交付周期长、产品焦点弱 | 销售和实施中使用边界清单 |
| 产品变成通用表单工具 | 差异化和合规价值弱 | 每个功能必须映射到 DHR 证据、生产执行、事务处理、DHR 汇总、放行、追溯或受控电子记录 |
| 事务/制程模型过轻 | 无法支撑真实过站和特殊业务动作 | P0 即建立制程配置、事务定义、事务实例和流程日志 |
| DHR 汇总被做成纯手工目录 | 审核效率低、实施成本高 | DHR 模板必须定义目录和自动聚合规则，人工拖入只做补充 |
| 客户误以为购买系统即自动完成企业整体合规 | 合规和信任风险 | 使用受控宣称和客户/供应商责任矩阵，明确软件自身受控设计与客户质量体系责任的边界 |
| 审计追踪/电子签名后补 | 重构成本高、受控基础弱 | 第一阶段先做流程与可信基座 |
| 模板过度灵活导致流程漂移 | 质量风险 | 模板生命周期、版本、生效、审批和变更历史必须受控 |
| 私有化部署支持成本高 | 交付成本上升 | 早期标准化部署、验证、备份和升级包 |
| 集成需求拖慢试点 | 销售和交付变慢 | 先用导入/人工引用和稳定 API 边界 |
| 伙伴并行设计业务模块不一致 | 后续返工 | 开发前发布领域模型、状态机、审计/签名 API 和模块边界 |

## 21. PRD 自检

### 产品/创始人视角

第一版可售卖范围较清晰，前提是持续聚焦 eDHR 证据、制程配置、事务驱动执行、DHR 汇总、放行审核和受控电子记录。差异化来自可产品化配置、快速试点、强可信基座和清晰边界。

### 最终用户视角

如果 Review by Exception 早期实现，系统能明显减少 QA 人工逐页检查。操作员侧需要足够顺手的扫码、拍照、附件和签名交互，否则会被认为增加工作量。

### 质量/法规视角

本文档避免把“客户整体质量体系或法规合规”宣称为系统自动保证的结果，并将最终质量责任归于客户质量角色。同时，文档明确要求软件自身围绕 eDHR 预期用途进行受控设计，审计追踪、电子签名、流程日志、保存期限、验证/确认支持、放行证据和追溯均被列为核心需求。

### 工程视角

第一步开发可行，因为它把领域模型、审计、签名、流程、表单、制程、事务、DHR 汇总和受控记录存储先隔离出来，再让伙伴设计检验、记录本、物料、返工和行业模板。深度集成被延后，不会阻塞 MVP。

### 数据安全视角

默认建议私有化部署。当前安全需求足以支撑规划，但正式开发前仍需要单独的技术安全设计。

### 评分

当前 PRD 估算成熟度：90 / 100。

剩余 P1 风险：

- 尚未选择第一个目标行业或场景包，因此 Phase 2 模板深度保持开放。
- 电子签名的法律/技术实现需要专项评审。
- 部署架构和备份/恢复方案需要独立技术规格。
- 验证/确认支持资料包需要在客户试点前进一步成文。

## 22. P0 边界拍板与仍需确认的问题

### 22.1 已拍板的 P0 边界

| # | 问题 | 决策 | 理由 |
| --- | --- | --- | --- |
| 1 | P0 首个闭环是否只以批次生产为主？ | 是。SN 保留数据模型和基础查询。 | 批次覆盖约 80% 场景，SN 混合会让状态机膨胀 2-3 倍。 |
| 2 | 事务节点是否只做表单节点 + 条件分支？ | 是。返工配置/拆分配置作为 P1。 | P0 要先把“触发 -> 填表 -> 进入 DHR”这条路径跑通。 |
| 3 | 放行 P0 是否只做放行单模板 + 填报/审核？ | 是。放行事务作为 P1。 | 放行单本质是受控表单，审核流程引擎已能支撑。 |
| 4 | 检验 P0 是否先走表单记录进入 DHR？ | 是。不独立做检验作业。 | 复用表单 + 流程能力，P1 再拆出独立检验域。 |
| 5 | 记录本是否放到 P1？ | 是。P0 只保留 DHR 汇总补充证据接口。 | P0 已有附录 + 事务表单补充来源，记录本不急。 |
| 6 | AI 表单还原是否作为 P1？ | 是。不阻塞 P0。 | P0 先把结构化模板模型做扎实。 |
| 7 | P0 审核人规则是否先支持静态匹配？ | 是。P0 支持用户/角色/部门/组织；动态业务上下文如工厂质量负责人等放 P1。 | 静态匹配覆盖约 90% 审核场景。 |

### 22.2 仍需确认的问题

1. 第一批试点客户可接受的最低电子签名实现是什么？
2. 哪些集成对首个试点是必须的，哪些可以先用手工或导入？
3. 客户期望的归档包格式是什么：PDF、结构化 JSON/CSV、带附件 ZIP，还是全部需要？
4. 第一个 P1 场景包优先选 SN 执行、检验域、记录本、返工配置，还是放行事务？
