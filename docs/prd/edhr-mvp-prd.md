## 2. 产品摘要

### 2.1 一句话定义

面向国内医疗器械生产企业，建设一款以 2025 版医疗器械 GMP 为核心法规锚点的 eDHR 系统，用于管理批次、台套或序列号级别产品的生产记录、检验记录、追溯证据、审核放行和归档导出。

### 2.2 要解决的问题

大量医疗器械企业仍依赖纸质记录、Word/Excel、共享文件夹、ERP/MES/QMS/LIMS 零散数据来证明每一批、每一台或每一个序列号产品已经按照受控要求完成生产、检验、审核和放行。这会带来几个典型问题：

- 批生产记录填写慢、漏填多、复核成本高。
- QA 放行前需要人工跨纸质记录、检验报告、标签、偏差记录、ERP 数据逐项核对。
- 电子记录的数据完整性能力不足，修改原因、原值/新值、操作人、日期、审计追踪和电子签名证据不完整。
- 产品、批号、序列号、UDI、物料批号、关键部件、设备、工艺参数、检验结果、异常处理和放行结论之间的追溯链条断裂。
- 2025 版医疗器械 GMP 更明确强调信息化系统、电子记录/数据、软件确认、每批/台生产记录、检验记录、产品放行、追溯、偏差/不合格/返工和委托生产衔接。

### 2.3 产品方案

产品定位为专注 eDHR 的合规生产记录与放行证据系统，不做完整 MES，不做完整 QMS。

产品由六类能力组成：

- 合规可信基座：权限、审计追踪、电子签名、受控记录存储、工作流、数据完整性、备份/恢复要求、CSV/CSA 支持包。
- 受控配置中心：产品/规格型号、DHR 记录包、电子记录模板、模板版本、判定规则、基础主数据。
- eDHR 核心业务：DHR 实例、生产记录、检验记录、物料/部件追溯、设备/环境/工艺参数记录、标签/UDI 记录、异常挂接。
- 审核放行中心：Review by Exception、完整性检查、QA 审核、产品放行、归档导出。
- 集成与数据服务：ERP/MES/WMS/QMS/DMS/LIMS/设备接口边界，优先支持导入和轻量 API。
- 基础体验与运维：待办、查询、通知、基础看板、系统配置、操作日志和运维日志。

### 2.4 成功指标

| 指标 | 当前基线 | MVP 目标 | 测量方式 |
| --- | --- | --- | --- |
| 单批/单台 DHR 审核时间 | 试点前采样 | 试点场景降低 30% | 试点前后 QA 审核时间对比 |
| 放行前缺项识别 | 依赖人工 | 已配置必填项 100% 被系统识别 | 完整性检查日志 |
| 受控记录审计追踪覆盖率 | 通常不完整 | 受控新增、修改、删除、状态流转、签名、放行 100% 留痕 | 测试用例与运行抽样 |
| 电子签名证据完整性 | 通常不一致 | 签名人、时间、含义、对象、认证事件 100% 记录 | 签名记录检查 |
| DHR 归档包准备时间 | 人工整理 | 放行后 5 分钟内生成试点 DHR 包 | 产品日志与用户观察 |

## 3. 范围与边界

### 3.1 产品定位

产品管理的是“某一批、某一台或某一序列号产品实际如何被生产、检验、复核、放行和归档的证据链”。

推荐对外描述：

> 基于 2025 版医疗器械 GMP 的 eDHR 系统，支持电子生产记录、检验记录、追溯、例外审核、产品放行证据、审计追踪、电子签名和检查准备。

### 3.2 范围内

- 电子 DHR 模板及受控版本。
- 按产品、规格型号、版本配置 DHR 记录包。
- 按批号、台套、序列号、UDI、工单或生产批次生成 DHR 实例。
- 生产过程记录填写，包括扫码、附件、图片、签名和复核。
- 原材料、关键部件、中间产品、成品、批号、序列号和 UDI 相关追溯。
- 工艺参数、设备、工装、环境、清场、包装、标签确认等可配置记录。
- 进货检验、过程检验、成品检验在 DHR 证据链中的记录或挂接。
- 偏差、不合格、返工、返修、让步、紧急放行、CAPA 编号挂接；完整 QMS 流程不在 MVP 内。
- Review by Exception：缺项、未复核、未签名、超限、异常未关闭、模板/版本不一致、放行阻断项。
- 产品放行流程和放行证据固化。
- DHR 归档包导出。
- 合规基座：RBAC、审计追踪、电子签名、记录保存、备份/恢复要求、工作流状态机、CSV/CSA 支持包。

### 3.3 范围外

- 不做完整 MES：高级排产、派工、OEE、产能计划、复杂线边仓、现场绩效管理不在 MVP 内。
- 不做完整 QMS：CAPA、投诉、供应商质量、内审、管理评审、文控、培训、变更控制的完整生命周期不在 MVP 内。
- 不做完整 WMS/ERP：库存金额、采购、收货、货位优化、成本核算、库存事务不在 MVP 内。
- 不做完整 LIMS：样品生命周期、试剂/培养基管理、仪器管理、OOS/OOT 调查、实验室排程不在 MVP 内。
- 不做完整 UDI 平台：监管数据库上传、全量标签设计不在 MVP 内；MVP 可记录和核对 UDI/PI。
- 不做完整 DMS：文件起草、审批、分发、培训联动、作废副本控制不在 MVP 内；MVP 可引用受控文件编号和版本。
- 不自动判定企业是否符合 GMP。
- 不替代客户质量负责人或授权放行人员的最终判断。

### 3.4 合规宣称边界

允许的产品宣称：

- 支持 2025 版医疗器械 GMP 语境下的电子生产记录和检验记录管理。
- 支持电子记录的数据完整性控制、审计追踪、电子签名、追溯和放行证据。
- 帮助质量团队准备 DHR 证据包并进行例外审核。
- 提供供应商侧 CSV/CSA 验证支持材料。

需要避免的过度宣称：

- 客户购买或使用系统后，客户企业整体 GMP 合规状态自动成立。
- 系统自动判断产品是否可以放行。
- 系统替代质量部门审核或授权放行人员判断。
- 替代 QMS/MES/ERP/LIMS/DMS。

需要明确的是：软件自身必须在已声明的 eDHR 预期用途范围内进行合规设计，并提供数据完整性、权限、审计追踪、电子签名、记录保存、追溯、放行证据和验证支持能力。这里限制的是对“客户整体 GMP 合规结果”的过度承诺，不是否定软件自身应满足适用合规要求。

## 4. 目标用户与购买角色

| 角色 | 任务 | 痛点 | 成功信号 | 购买影响 |
| --- | --- | --- | --- | --- |
| 生产操作员 | 按要求完成生产记录 | 纸质记录影响操作、重复填写、怕漏填错填 | 录入快、可扫码、必填清楚 | 中 |
| 生产主管 | 跟踪 DHR 进度并纠正缺口 | QA 审核前难以及时发现记录不完整 | 实时看到 DHR 状态和待办 | 中 |
| QA 审核员 | 审核批次/台套证据并支持放行 | 逐页人工检查慢且容易漏 | 例外审核、缺项清楚、证据完整 | 高 |
| 放行审核人 | 确认放行条件并签发放行 | 需要确信生产、检验、异常、标签、UDI 均已闭环 | 放行清单和证据可追溯 | 高 |
| 质量负责人 | 保持记录合规和检查准备状态 | 纸电混合证据脆弱 | 审计追踪、签名、归档、检索可靠 | 高 |
| IT/CSV 负责人 | 部署并确认 GxP 相关系统 | 供应商验证资料不完整 | 系统说明、权限矩阵、测试脚本、发布记录齐全 | 高 |
| 工艺/工业工程师 | 将工艺要求转成受控执行记录 | 模板漂移、字段规则难维护 | 模板版本和规则可控 | 中 |
| 企业负责人 | 降低合规风险和质量成本 | 重型系统贵且慢 | 快速试点、审核效率提升、范围受控 | 高 |

初始目标客户：

- 国内二类、三类医疗器械生产企业。
- 生产记录纸质化严重、放行审核耗时长、追溯压力较大的企业。
- 面临新版 GMP、客户审计、注册人制度或委托生产协同压力的企业。
- 暂时不想上重型 MES，但需要快速电子化 DHR 的企业。

## 5. 市场与竞品环境

| 替代方案 | 用户用途 | 优势 | 缺口 | 我们的差异化 |
| --- | --- | --- | --- | --- |
| 纸质记录 + Word/Excel + 共享文件夹 | 当前 DHR 记录和审核 | 熟悉、便宜、灵活 | 审计追踪弱、审核慢、追溯差 | 受控电子记录链和例外审核 |
| 完整 MES | 生产执行和现场控制 | 能力强、集成深 | 贵、周期长、范围大 | 专注 eDHR，实施更轻 |
| 完整 QMS | CAPA、偏差、变更等质量流程 | 质量流程治理强 | 不擅长生产记录执行 | 只挂接 QMS 事件，不吞 QMS |
| ERP/WMS | 工单、BOM、库存、批号 | 主数据权威 | 不适合 DHR 证据链 | 接入业务数据，固化 DHR 证据 |
| LIMS | 样品和检验结果 | 实验室控制强 | 不覆盖生产放行证据 | 将检验证据纳入 DHR |
| 国内 eDHR/eBR 厂商如冠骋 | 医疗器械电子批记录和快速车间数字化 | 已验证国内需求 | 可能向轻 MES 扩展 | 更强调合规基座产品化和边界清晰 |
| 国际 eDHR 厂商 | eDHR、电子记录、质量/制造执行 | 成熟度高 | 成本高、本地 GMP 2025 适配和交付不一定轻 | 中国 GMP 2025 优先、本地交付和私有化友好 |

待验证假设：

- 国内早期客户更可能接受私有化部署或专有云，而不是公有 SaaS 存放真实生产质量记录。
- 如果 ERP/MES/QMS 集成边界清晰，客户会接受聚焦 eDHR 的独立产品。
- CSV/CSA 支持包会显著提升销售和试点转化。

### 6.1 分层架构

```text

待办、查询、通知、看板、系统配置、操作日志、运维日志

第 4 层：集成与数据服务
```

| 基础看板 | DHR 状态、完成率、放行周期、异常数量、归档数量 | P1 |
| 通知提醒 | 任务分配、退回、异常、审批、放行完成提醒 | P1 |
| 平板/移动端适配 | 现场可用的表单、扫码、拍照、签名交互 | P1 |

## 7. MVP 策略与第一步开发范围

### 7.1 MVP 主线

第一版可销售 MVP 应跑通这一条主线：

```text
受控模板版本
-> 生成 DHR 实例
-> 执行生产/检验记录
-> 建立物料/序列追溯
-> 挂接异常
-> QA 例外审核
-> 放行签名
-> DHR 归档导出
```

### 7.2 创始人单人优先开发范围

第一步建议先开发“合规可信基座 + 产品内核”，因为这部分不强依赖具体生产业务调研，并且会被后续业务模块复用。

Phase 0 范围：

- 核心领域对象和生命周期状态。
- 组织、用户、角色、权限模型。
- 审计追踪服务。
- 电子签名服务。
- 受控记录存储模型。
- 工作流/状态机模型。
- 模板与记录实例关系。
- DHR 实例聚合模型。
- 最小完整性检查和放行阻断框架。
- 架构与 API 边界文档。

暂时不要先做：

- 深度生产路线建模。
- 完整检验项目库。
- 复杂物料平衡。
- 完整偏差/CAPA 流程。
- ERP/MES/WMS/QMS/DMS 深度集成。
- 设备/仪器联网。
- 大屏和复杂看板。

### 7.3 MVP 非目标

MVP 不能变成通用低代码表单平台。每个功能都必须能映射到 DHR 证据、2025 版 GMP 电子记录要求、放行审核、追溯或软件确认/验证准备。

## 8. 核心场景

### 场景 1：配置受控 DHR 模板

- 触发：工艺/质量团队需要电子化某个生产或检验记录。
- 角色：工艺工程师、QA 审核员、系统管理员。
- 当前方式：Word/Excel 表单在 DMS 受控后打印或手工填写。
- 目标流程：创建模板草稿，配置字段/规则/签名，审核批准并生效。
- 输出证据：模板版本、批准/签名记录、审计追踪、生效状态。
- 异常路径：模板退回、版本作废、生效后变更。
- 验收标准：
  - 已生效模板不能被静默编辑。
  - 新 DHR 实例必须绑定一个生效模板版本。
  - 已作废模板不能用于新 DHR，除非有受控例外。

### 场景 2：生成并执行 DHR 实例

- 触发：工单、批次、台套或序列号进入生产。
- 角色：生产主管、操作员、QA。
- 当前方式：人工组装纸质 DHR 包或 Excel 记录。
- 目标流程：按产品/型号/模板版本生成 DHR，分配任务，填写记录，采集物料、参数、设备和人员数据。
- 输出证据：DHR 实例、记录实例、字段值、附件、签名、审计事件。
- 异常路径：缺字段、物料批号错误、超限值、退回复填、受控补录。
- 验收标准：
  - 必填字段未完成时不能完成记录。
  - 超限值产生异常标记。
  - 签名后修改必须进入受控更正流程并产生审计追踪。

### 场景 3：采集检验证据

- 触发：DHR 记录包要求进货、过程或成品检验。
- 角色：检验员、复核员、QA。
- 当前方式：独立检验单或检验报告附在批记录中。
- 目标流程：记录检验项目、设备、结果、日期、检验员、复核人、结论和报告附件。
- 输出证据：与 DHR 关联的检验记录。
- 异常路径：不合格结果、复检、委托检验证书。
- 验收标准：
  - 检验记录可包含产品/原材料信息、检验项目、设备、结果、日期、检验员和复核人。
  - 不合格结果在未完成处置前阻断正常放行，除非授权人员进行受控处理。

### 场景 4：QA 例外审核

- 触发：DHR 提交 QA 审核或放行审核。
- 角色：QA 审核员、放行审核人。
- 当前方式：人工逐页审核。
- 目标流程：系统自动暴露缺项、未签名、超限、异常未关闭、版本不一致，QA 钻取证据。
- 输出证据：审核结论、审核意见、签名、问题列表。
- 异常路径：退回生产、条件放行、紧急放行、记录更正。
- 验收标准：
  - 系统列出全部配置的放行阻断项。
  - QA 只有在阻断项关闭，或授权角色带原因和签名覆盖后，才能继续放行流程。

### 场景 5：产品放行和归档

- 触发：DHR 证据准备进入放行决定。
- 角色：放行审核人、质量负责人、归档负责人。
- 当前方式：人工整理放行决定和 DHR 证据包。
- 目标流程：确认放行清单，签署放行决定，冻结 DHR，生成归档/导出包。
- 输出证据：放行决定、放行签名、只读 DHR 状态、DHR 归档包。
- 异常路径：放行后更正/补充、召回追溯、客户审计请求。
- 验收标准：
  - 已放行 DHR 除受控更正/补充流程外不可修改。
  - 归档包包含 DHR 记录、放行决定、签名、附件、审计追踪和元数据。

### 场景 6：委托生产证据衔接

- 触发：注册人/备案人与受托生产企业共同管理生产和放行证据。
- 角色：委托方 QA、受托方生产/QA、放行审核人。
- 当前方式：邮件、PDF 扫描件、人工证据传递。
- 目标流程：生产放行证据和上市放行证据可分离但可追溯关联。
- 输出证据：生产放行记录、上市放行记录、生产过程和质量控制证据。
- 异常路径：生产偏差、受托方变更、证据传递延迟。
- 验收标准：
  - 权限和数据访问可区分委托方、受托方职责。
  - 在配置启用时，系统可分别建模生产放行和上市放行。

## 9. 用户故事

1. 作为系统管理员，我希望配置组织、用户、角色和权限，以便只有授权用户可以创建、修改、复核、签名、放行、导出或管理受控记录。
2. 作为工艺工程师，我希望创建带版本控制的 DHR 模板，以便每个 DHR 实例都基于已批准的记录要求。
3. 作为 QA 审核员，我希望模板审批和变更都有审计追踪，以便检查时能证明模板受控。
4. 作为生产主管，我希望按产品、批号、台套、序列号或工单生成 DHR，以便围绕产品证据链组织执行记录。
5. 作为操作员，我希望系统提供引导式电子记录和必填/范围校验，以便生产时正确完成记录。
6. 作为操作员，我希望扫码录入物料批号或序列号，以便减少重复手填并形成追溯。
7. 作为检验员，我希望记录检验项目、设备、结果、日期和复核人，以便检验证据完整可追溯。
8. 作为复核/审核人员，我希望使用带签名含义的电子签名，以便复核和批准证据清晰。
9. 作为 QA 审核员，我希望按例外审核 DHR，以便优先处理缺失、异常、未签名或未关闭事项。
10. 作为放行审核人，我希望基于配置的放行条件查看放行清单，以便作出并签署放行决定。
11. 作为质量负责人，我希望每次受控记录修改都保留原值、新值、原因、操作人和时间，以便满足电子记录数据完整性预期。
12. 作为 IT/CSV 负责人，我希望获得供应商验证支持资料，以便完成系统确认/验证。
13. 作为客户审计或检查准备人员，我希望导出完整 DHR 归档包，以便快速提供证据。
14. 作为业务模块设计伙伴，我希望有明确 API 和模块边界，以便设计生产、检验、追溯模块时不用重复实现审计、签名和工作流。

## 12. AI 需求

AI 不在 MVP 范围内。

如果未来加入 AI/OCR/文档理解能力，AI 不得静默修改受控记录或最终放行决定。所有 AI 建议必须有来源引用、置信度或不确定性提示、人工确认、模型/提示词版本追踪，并且除非合同明确允许，不得用于模型训练。

| Signature | 电子签名证据 | 签名人、含义、对象、时间、认证事件、哈希 | 有效/已失效/被替代 |
| AuditEvent | 审计事件 | 操作人、动作、对象、原值/新值、原因、时间、来源 | 不可变 |
| ReleaseDecision | 放行证据 | 审核人、清单、结论、条件、签名 | 待处理/已放行/已拒绝/条件放行 |
| ArchivePackage | 导出证据包 | 包 ID、DHR ID、文件清单、哈希、导出人 | 已生成/被替代 |

### 13.2 状态机

模板：

```text
草稿 -> 审核中 -> 已生效 -> 已作废
草稿 -> 已作废
审核中 -> 已退回 -> 草稿
```

DHR 实例：

```text
已创建 -> 执行中 -> 审核中 -> 待放行 -> 已放行 -> 已归档
审核中 -> 已退回 -> 执行中
待放行 -> 已退回 -> 审核中
已放行 -> 更正待处理 -> 更正已批准 -> 已放行
```

记录实例：

```text
打开 -> 已提交 -> 已复核
已提交 -> 已退回 -> 打开
已复核 -> 更正待处理 -> 已更正 -> 已复核
打开/已提交 -> 已作废
```

异常链接：

```text
打开 -> 处理中 -> 已关闭
打开/处理中 -> 放行豁免
```

放行决定：

```text
待处理 -> 已批准
待处理 -> 已拒绝
待处理 -> 条件批准
已批准/条件批准 -> 已归档
```

## 15. 部署与数据安全

| 模式 | 目标客户 | 数据处理 | 优势 | 风险 | MVP 建议 |
| --- | --- | --- | --- | --- | --- |
| 私有化部署 | 对生产质量数据敏感的医疗器械企业 | 客户控制基础设施和数据库 | 信任度最高，符合保守客户预期 | 交付和支持更重 | MVP 主模式 |
| 专有云/单租户云 | 接受托管服务的中型客户 | 每客户独立数据库/存储 | 更新和支持更快 | 需要安全评审 | 设计兼容，非第一依赖 |
| 公有 SaaS | 演示、沙箱、低风险试用 | 不建议存真实生产记录，除非合同明确 | 低摩擦 | 对真实 DHR 信任低 | 仅作演示 |

安全基线：

- 所有受控操作服务端鉴权。
- 密码和会话策略可配置。
- 签名、放行等敏感操作支持二次认证。
- 登录、失败登录、权限变更、导出、接口导入、管理员操作均留日志。
- 传输加密；静态加密根据部署环境支持并写入部署指南。
- 提供备份/恢复流程和恢复测试证据模板。

## 17. 商业模式与包装

首个可售卖包装建议：

- 私有化部署或专有云选项。
- 核心 eDHR 包：模板/版本、DHR 实例、生产/检验记录、追溯、例外审核、放行、归档导出。
- 合规基座：RBAC、审计追踪、电子签名、受控存储、工作流、验证支持包。
- 实施包：一个试点产品族、3-5 个记录模板、一个放行流程、一个归档包格式。
- 年度维护和验证支持续费。

避免纯项目制。实施工作应限制在模板、主数据、角色、工作流配置和少量接口范围内。

## 19. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 范围蔓延成 MES/QMS/WMS/LIMS | 交付周期长、产品焦点弱 | 销售和实施中使用边界清单 |
| 产品变成通用表单工具 | 差异化和合规价值弱 | 每个功能必须映射到 DHR 证据、放行、追溯或 GMP 2025 数据完整性 |
| 客户误以为购买系统即自动完成企业整体 GMP 合规 | 合规和信任风险 | 使用受控宣称和客户/供应商责任矩阵，明确软件自身合规设计与客户质量体系责任的边界 |
| 审计追踪/电子签名后补 | 重构成本高、合规基础弱 | 第一阶段先做合规可信基座 |
| 模板过度灵活导致流程漂移 | 质量风险 | 模板生命周期、版本、生效、审批和变更历史必须受控 |
| 私有化部署支持成本高 | 交付成本上升 | 早期标准化部署、验证、备份和升级包 |
| 集成需求拖慢试点 | 销售和交付变慢 | 先用导入/人工引用和稳定 API 边界 |
| 伙伴并行设计业务模块不一致 | 后续返工 | 开发前发布领域模型、状态机、审计/签名 API 和模块边界 |

## 21. PRD 自检

### 产品/创始人视角

第一版可售卖范围较清晰，前提是持续聚焦 eDHR 证据、放行审核和 GMP 2025 电子记录控制。差异化来自中国 GMP 2025 优先、快速试点、强合规基座和清晰边界。

### 最终用户视角

如果 Review by Exception 早期实现，系统能明显减少 QA 人工逐页检查。操作员侧需要足够顺手的扫码、拍照、附件和签名交互，否则会被认为增加工作量。

### 质量/法规视角

本文档避免把“客户整体 GMP 合规”宣称为系统自动保证的结果，并将最终质量责任归于客户质量角色。同时，文档明确要求软件自身围绕 eDHR 预期用途进行合规设计，审计追踪、电子签名、保存期限、软件确认支持、放行证据和追溯均被列为核心需求。

### 工程视角

第一步开发可行，因为它把领域模型、审计、签名、工作流和受控记录存储先隔离出来，再让伙伴设计具体生产业务模块。深度集成被延后，不会阻塞 MVP。

### 数据安全视角

默认建议私有化部署。当前安全需求足以支撑规划，但正式开发前仍需要单独的技术安全设计。

### 评分

当前 PRD 估算成熟度：88 / 100。

剩余 P1 风险：

- 尚未选择第一个目标医疗器械细分场景，因此 Phase 2 模板深度保持开放。
- 电子签名的法律/技术实现需要专项评审。
- 部署架构和备份/恢复方案需要独立技术规格。
- CSV/CSA 支持资料包需要在客户试点前进一步成文。
