## 2. 产品摘要

### 2.1 一句话定义

商业级可定制 eDHR 商用版是一套面向高监管制造和复杂质量记录场景的生产记录与 DHR 证据链系统，用于将产品从基础建模、表单/DHR/放行模板、制程配置、事务配置、生产执行、表单填报与审核、DHR 自动汇总、放行审核到归档追溯的全过程电子化、结构化、流程化和可审计化。

### 2.2 产品愿景

成为企业在“不上重型 MES、不重做 QMS、不陷入纯项目低代码”的前提下，快速搭建可定制 eDHR 业务闭环的专业平台。产品要能按客户行业、产品形态和工艺路线配置，但底层对象、流程、审计、签名、DHR 汇总和追溯模型保持产品化。

### 2.3 价值主张

- 对生产：按制程配置引导过站、填表、触发事务，减少漏填、错填、重复录入和纸质流转。
- 对质量：用 DHR 汇总模板、完整性检查和 Review by Exception 替代逐页人工翻查，提高审核放行效率。
- 对工艺/实施：通过结构化表单、DHR 模板、放行单模板、工艺路线、制程配置和事务配置实现可复用交付。
- 对法规/客户审计：形成可追溯、可检索、可导出的 DHR 证据包，并保留审计追踪、电子签名和流程日志。
- 对 IT/验证：提供系统说明、权限矩阵、审计追踪说明、电子签名说明、测试脚本和发布记录，降低受控系统确认/验证负担。
- 对管理层：以低于重型 MES 的成本和周期，优先解决生产质量记录、DHR 汇总和放行证据问题。

### 2.4 商用成功指标

| 指标 | 商用目标 | 测量方式 |
| --- | --- | --- |
| 试点上线周期 | 单产品族/单产线 4-8 周内完成首批 DHR 上线 | 项目里程碑 |
| QA 审核效率 | 试点产品 DHR 审核时间降低 30%-50% | 上线前后采样 |

## 3. 产品定位与边界

### 3.1 产品定位

本产品是 eDHR 系统，不是完整 MES、QMS、ERP、WMS、LIMS、DMS、UDI 平台或通用低代码平台。它的核心对象是可配置 DHR 证据链：

```text
基础主数据
-> 表单/DHR/放行模板
-> 工艺路线与制程配置
-> 事务定义与事务配置
-> 工单/批次/SN
-> 工序执行、表单填报、事务处理
-> DHR 实例和 DHR 汇总
-> 审核、变更、作废、放行
-> 归档导出
-> 追溯查询
```

### 3.2 对外推荐描述

> 面向高监管制造和复杂质量记录场景的商业级 eDHR 系统，支持基础建模、结构化表单、DHR 模板、制程配置、事务驱动生产执行、DHR 自动汇总、审核放行、审计追踪、电子签名、归档导出和追溯查询。医疗器械 GMP、CSV/CSA 等能力可作为行业实施包启用。

### 3.3 明确不做

- 不将“客户使用本系统”表述为“客户整体质量体系或法规合规已经由系统自动保证”。系统自身必须按声明的预期用途进行受控设计、验证支持和受控交付。
- 不自动替代质量负责人或授权放行人员判断。
- 不做无限自由的 Excel 画布式表单设计器。P0 做结构化字段、表格、附件、签名块、审核块和校验规则；P1 再做 AI 辅助表单还原草稿。
- 不做完整生产排产、OEE、APS、复杂派工。
- 不做完整库存、采购、财务、成本核算。
- 不做完整 CAPA、投诉、供应商质量、变更控制和内审管理。
- 不做完整实验室 LIMS。
- 不做完整文控 DMS。
- 不做完整 UDI 申报平台。

### 3.4 和其他系统的关系

| 系统 | eDHR 关系 |
| --- | --- |
| ERP | 接收工单、产品、BOM、批号、计划数量；返回 DHR/放行状态 |
| MES | 可接收生产上下文；不替代 MES 的排产、派工和设备执行 |
| WMS | 校验物料批号、状态、效期；不替代库存事务 |
| QMS | 关联偏差、不合格、CAPA、变更编号；不替代 QMS 流程 |
| DMS | 引用 SOP、工艺规程、检验规程编号和版本；不替代文控生命周期 |
| LIMS | 接收或挂接检验结果、报告、证书；不替代样品和实验室全流程 |
| UDI 平台 | 记录和核对 UDI/PI；不替代监管数据库申报 |
| 集成中心/iPaaS | P0 不做可视化连接流；先做导入、受控引用、OpenAPI 和接口审计，P2 再考虑连接流 |

## 4. 目标客户与用户角色

### 4.1 首选客户画像

- 有批次、台套、SN、工单、物料、设备、检验、放行证据链要求的制造企业。
- 纸质记录、Word/Excel、共享文件夹或局部系统记录较多，DHR 汇总和审核依赖人工。
- 需要按不同产品、工艺路线、工序、表单和放行规则快速定制执行流程。
- 暂时不想实施重型 MES，也不想把 QMS/WMS/LIMS/DMS 全部重做，但需要快速提升生产质量记录闭环。
- 医疗器械可作为首个高监管验证场景；后续行业进入应通过模板包、事务模板和实施配置适配。

### 4.2 适合优先切入的场景类型

| 场景类型 | 适配原因 | 需要注意 |
| --- | --- | --- |
| 批次型生产 | DHR 汇总、批记录、检验和放行路径清晰 | P0 优先，以批次闭环建立产品内核 |
| 批次 + SN 混合 | 需要序列追溯和关键部件记录 | P0 保留模型，P1 深化 SN 执行 |
| 装配/离散制造 | 设备、部件、调试、检验表单多 | 需要较好的表单、工序和事务配置 |
| 高监管制造 | 审计追踪、签名、放行、保存期和验证资料价值高 | 行业法规不能写死在平台核心里 |
| 委外/多组织协同 | 生产证据和放行证据跨组织流转 | P2 专项设计权限、责任和数据边界 |

### 4.3 用户角色

| 角色 | 核心任务 | 关键诉求 |
| --- | --- | --- |
| 生产操作员 | 按步骤执行并填写记录 | 简单、快速、少打字、少返工 |
| 生产主管 | 查看进度、处理退回、监督缺项 | 实时状态、异常提醒 |
| 工艺工程师 | 配置工序、模板、规则 | 版本受控、可复用、少定制 |
| 检验员 | 填写/导入检验结果 | 项目清楚、设备/结果可追溯 |
| QA 审核员 | 审核 DHR 和异常 | 例外审核、证据可钻取 |
| 放行审核人 | 作出放行决定 | 放行条件清楚、责任边界清楚 |
| 质量负责人 | 管理质量证据和检查准备 | 审计追踪、签名、归档、检索 |
| IT/验证负责人 | 部署、运维、验证/确认 | 供应商验证包、权限矩阵、安全说明 |
| 实施/配置顾问 | 建模、模板、流程、制程和事务配置 | 快速复用、少定制代码、配置可追溯 |
| 委外协同 QA | 查看或共享跨组织生产放行证据 | 权限隔离、责任边界清楚、证据传递高效 |

## 7. 核心业务流程

系统流程不能被设计成几个硬编码审批流。商用版至少需要审核流程引擎、表单实例流程、事务流程引擎、DHR 汇总/放行流程、变更/作废流程五类能力，并统一运行日志、任务、消息、转办和审计追踪。

### 7.1 基础建模到制程发布流程

```text
维护工厂/车间/产线/设备/SOP/工序
-> 维护产品家族/产品/产品版本
-> 创建工艺路线
-> 配置制程：产品 + 路线 + DHR 模板 + 放行单模板 + 工序表单/SOP/权限
-> 绑定全局/按需事务
-> 发布制程版本
-> 生产执行按已发布制程版本运行
```

关键规则：

- 已发布制程版本不可静默修改。
- 批次执行必须绑定来源制程版本；SN 在 P0 只保留数据模型和基础查询。
- 制程变更不自动改变已创建的批次/DHR，除非走受控变更或补充流程。

### 7.2 模板受控与表单实例流程

```text
创建表单/DHR/放行单模板草稿
-> 配置字段、表格、附件、扫码、签名块、审核块和校验规则
-> 配置表单实例流程：填报节点、审核节点、字段权限、审核组织/人员
-> 模板审批
-> 发布受控版本
-> 被制程配置、事务节点或 DHR 模板引用
```

关键规则：

- P0 不做 Excel 式自由画布，而做结构化组件和字段语义。
- 表单流程控制“谁填、谁审、每个节点能看或改哪些字段”。
- AI 表单还原只能在 P1 生成模板草稿，必须人工确认和审批后发布。

### 7.3 事务驱动生产执行流程

```text
接收或创建工单/批次
-> 按制程版本生成批次执行和 DHR 实例
-> 工序开始
-> 展示工序待填表单、SOP、权限、可触发事务
-> 操作员填报表单并提交
-> 表单按表单实例流程审核
-> 特殊情况触发全局/按需事务
-> 事务流程按表单节点、条件分支和处理任务运行
-> 工序完成或被阻断
-> 进入下一工序
```

关键规则：

- 事务配置 + 制程配置共同决定生产执行过站时用户看到哪些事务、填哪些表单、形成哪些 DHR 候选证据。
- 事务实例必须保存来源事务版本，并关联批次、工序、生产/返工场景；SN 事务入口作为 P1 执行能力。
- 我的事务、批次事务只是 P0 事务实例的不同入口；SN 事务作为 P1 扩展。

### 7.4 DHR 汇总、审核与放行流程

```text
生产执行产生表单实例、事务表单、返工表单、附录表单
-> DHR 模板按目录自动聚合证据
-> 用户从候选证据补充或整理目录
-> 系统执行完整性检查
-> DHR 汇总提交审核
-> DHR 审核通过后按制程配置生成放行单
-> 放行单填报并审核
-> 放行签名
-> DHR 冻结归档
-> 导出 DHR 归档包和追溯证据
```

DHR 汇总不是从空目录完全手工拖拽。正确产品逻辑是 DHR 模板先定义目录和必需证据，系统自动聚合；人工拖入表单实例只是用于附录、事务、返工、记录本等补充证据。

### 7.5 变更、作废与干预流程

```text
用户发起表单/DHR/模板/配置变更或作废
-> 填写原因、影响范围、前后对比
-> 按对象类型命中变更/作废审核流程
-> 审核、退回或转办
-> 必要时触发签名失效、重新复核、DHR 重新汇总或放行后更正
-> 全过程进入审计追踪和流程日志
```

流程干预能力 P0 只做最小转办；撤回、强制跳转、补偿重试、超时升级、消息节点和流程建模追溯放入 P1/P2。

### 10.2 关键实体

| 实体 | 说明 | 商用版关注点 |
| --- | --- | --- |
| WorkflowVersion | 流程版本 | 审核流程、表单流程、DHR 汇总审核、变更/作废、放行审核复用 |
| WorkflowInstance | 流程运行实例 | 任务、退回、转办、日志、干预和消息记录 |
| FormTemplateVersion | 受控表单模板版本 | 生效后不可变，历史表单实例保持原版本 |
| DhrTemplateVersion | 受控 DHR 模板版本 | 定义目录、必需证据、自动聚合和补充证据规则 |
| ReleaseFormTemplateVersion | 受控放行单模板版本 | DHR 汇总后自动生成放行单 |
| ProcessVersion | 制程配置版本 | 绑定产品、路线、DHR 模板、放行单、工序表单、SOP、权限和事务 |
| TxnVersion | 事务定义版本 | 表单节点、条件分支、处理任务、消息预留、子事务预留 |
| ProductionBatch | 批次执行聚合 | 工单、批次、SN 轻量模型、工序状态、制程版本 |
| OperationExecution | 工序执行实例 | 过站、待填表单、可触发事务、阻断原因 |
| TxnInstance | 事务运行实例 | 关联批次、预留 SN、工序、事务版本，过程证据进入 DHR 候选 |
| FormInstance | 表单执行实例 | 状态机、字段权限、签名、退回、更正、来源对象 |
| DhrInstance | DHR 主聚合根 | 支持批次、台套、SN、工单混合结构 |
| DhrSummaryVersion | DHR 汇总版本 | 按 DHR 模板自动聚合并允许候选证据补充 |
| Signature | 电子签名证据 | 含义、认证、对象快照、失效规则 |
| AuditEvent | 审计事件 | 不可变、可检索、可导出 |
| ExceptionLink | 异常/质量事件链接 | 只挂接或轻量状态同步，不做完整 QMS |
| ReleaseFormInstance | 放行单实例 | 与放行审核流程、签名、证据快照绑定 |
| ArchivePackage | 归档包 | 可复现、带校验、含证据清单 |

## 14. 集成策略

### 14.1 总原则

- 商用版必须能独立运行。
- 集成增强效率，但不应成为首个客户上线的必要前提。
- 所有接口导入数据都必须校验、映射、留痕。
- 外部系统数据进入 DHR 后，应形成受控证据或受控引用。

### 14.2 集成优先级

| 优先级 | 集成 | 原因 |
| --- | --- | --- |
| P0 | CSV/Excel 导入、基础 API | 支持快速试点和私有化交付 |

| P1 | QMS 事件编号和状态 | 放行审核需要异常闭环信息 |

## 16. AI 路线边界

AI 不属于 v1.0 商用必需能力。

未来可选能力：

- AI/OCR 辅助历史纸质、Word、Excel、PDF、图片表单还原为结构化模板草稿。
- AI 辅助检查 DHR 缺项和证据分类。
- AI 辅助生成验证测试草稿或配置建议。

严格限制：

- AI 不得静默修改受控记录。
- AI 不得作出最终放行决定。
- AI 建议必须有来源、版本、置信度或不确定性提示。
- AI 输出必须人工确认后才能进入受控记录。
- AI 不能直接发布模板，也不能绕过模板审批、表单流程或签名规则。
- 客户数据是否进入模型训练必须由合同明确约定，默认不得用于训练。

## 18. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 商用范围过宽变成轻 MES | 交付变慢、竞争变复杂 | 保持 eDHR 证据链优先，MES 能力只做必要上下文 |
| 客户要求无限表单定制 | 产品化失败 | 建立结构化组件、模板包和 AI 草稿边界，不做 Excel 画布复刻 |
| 事务/制程配置模型过轻 | 无法驱动真实生产执行 | P0 即建立制程配置、事务定义、事务实例和流程日志 |
| DHR 汇总被误做成纯手工目录 | 审核闭环弱、实施成本高 | DHR 模板必须定义目录和自动聚合规则，手工拖入只做补充 |
| 电子签名方案被客户/审计质疑 | 上线受阻 | 早期完成法律/质量/IT 联合评审，提供签名说明和 SOP 建议 |
| 审计追踪不可变性不足 | 合规风险 | 审计事件独立模型、无普通删除入口、导出可核验 |
| 私有化部署支持成本高 | 毛利下降 | 标准部署包、升级包、监控和备份流程 |
| 接口项目化过重 | 实施成本失控 | 先标准 API/导入，再做客户接口适配 |
| 跨组织协同权限设计不严 | 数据泄露和责任混乱 | v1.0 预留模型，v2.0 专项设计 |
| 产品宣称过度 | 法规和销售风险 | 明确区分“软件自身按预期用途受控设计”和“客户整体质量体系/法规责任”，不宣称系统自动保证客户整体合规或自动放行 |

## 19. PRD 自检

### 产品/创始人视角

正式商用版边界清晰：以 eDHR 证据链、制程配置、事务驱动执行、DHR 汇总、审核放行和受控电子记录为主，不把 MES/QMS/WMS/LIMS/DMS/iPaaS 吞进来。商业包装需要在“标准产品 + 模板/事务包 + 有边界实施”上保持纪律。

### 最终用户视角

生产操作员需要低摩擦输入，QA 需要例外审核和证据钻取，质量负责人需要审计追踪、签名、归档和检查包。商用版如果只强调合规而忽略现场体验，会影响落地。

### 质量/法规视角

PRD 将医疗器械 GMP 2025 的电子记录、软件确认、生产记录、检验记录、追溯和放行作为高监管实施参考，而不是平台总边界。文档要求软件自身按预期用途进行受控设计，同时避免把客户整体质量体系合规或最终产品放行责任宣称为系统自动完成。

### 工程视角

v1.0 应先保证核心闭环和底座稳定，接口、行业包、跨组织协同和设备采集按路线图推进。最关键工程风险是流程引擎、表单模型、制程配置、事务运行、DHR 汇总、审计追踪、电子签名和归档包的模型设计。

### 数据安全/运维视角

私有化部署优先符合高监管制造客户对生产质量数据的信任预期，但需要标准部署、备份、恢复、升级和日志方案，否则交付成本会很快上升。

### 评分

当前 PRD 成熟度估算：90 / 100。

剩余 P1 风险：

- 首个行业或场景模板包尚未确定。
- 电子签名法律技术方案需要专项设计。
- 私有化部署架构、安全基线和运维 SLA 需要技术规格。
- 验证/确认文档包需要单独落地为可交付模板。

## 20. P0 边界拍板与待创始团队确认事项

### 20.1 已拍板的 P0 边界

| # | 问题 | 决策 | 理由 |
| --- | --- | --- | --- |
| 1 | P0 首个闭环是否只以批次生产为主？ | 是。SN 保留数据模型和基础查询。 | 批次覆盖约 80% 场景，SN 混合会让状态机膨胀 2-3 倍。 |
| 2 | 事务节点是否只做表单节点 + 条件分支？ | 是。返工配置/拆分配置作为 P1。 | P0 要先把“触发 -> 填表 -> 进入 DHR”这条路径跑通。 |
| 3 | 放行 P0 是否只做放行单模板 + 填报/审核？ | 是。放行事务作为 P1。 | 放行单本质是受控表单，审核流程引擎已能支撑。 |
| 4 | 检验 P0 是否先走表单记录进入 DHR？ | 是。不独立做检验作业。 | 复用表单 + 流程能力，P1 再拆出独立检验域。 |
| 5 | 记录本是否放到 P1？ | 是。P0 只保留 DHR 汇总补充证据接口。 | P0 已有附录 + 事务表单补充来源，记录本不急。 |
| 6 | AI 表单还原是否作为 P1？ | 是。不阻塞 P0。 | P0 先把结构化模板模型做扎实。 |
| 7 | P0 审核人规则是否先支持静态匹配？ | 是。P0 支持用户/角色/部门/组织；动态业务上下文如工厂质量负责人等放 P1。 | 静态匹配覆盖约 90% 审核场景。 |

### 20.2 仍需创始团队确认

1. 第一商用版本命名与包装：Foundation/Professional/Enterprise 是否合适。
2. 私有化部署的技术栈、数据库、中间件和运维支持边界。
3. 电子签名最低实现和未来增强路径。
4. 第一个客户必须集成的系统清单。
5. 验证/确认支持是“仅提供文档模板”还是“提供辅助验证服务”。
6. 行业/场景模板包是否作为单独商业模块收费。
7. 第一个 P1 场景包优先选 SN 执行、检验域、记录本、返工配置，还是放行事务。

## 21. 剩余假设

- 首批客户主要在中国大陆市场。
- 客户真实生产质量记录更倾向私有化或专有云部署。
- 第一阶段先做 eDHR 合规基座和核心闭环，再让业务专家补充细分场景模板。
- 产品自身在高监管客户场景中可能作为 GxP 或质量相关信息化系统，需要被客户进行确认/验证。
