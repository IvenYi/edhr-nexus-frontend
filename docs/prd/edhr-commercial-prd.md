## 2. 产品摘要

### 2.1 一句话定义

医疗器械 eDHR 商用版是一套面向国内医疗器械生产企业的合规生产记录与放行证据系统，用于将产品从受控模板、生产执行、检验记录、追溯、异常处理、质量审核、产品放行到归档导出的完整 DHR 证据链电子化、结构化和可审计化。

### 2.2 产品愿景

成为国内医疗器械制造企业在“不上重型 MES、不重做 QMS”的前提下，快速实现 DHR 无纸化、放行审核提效、GMP 2025 电子记录数据完整性和检查准备的专业系统。

### 2.3 价值主张

- 对生产：减少漏填、错填、重复录入和纸质流转。
- 对质量：用 Review by Exception 替代逐页人工翻查，提高放行审核效率。
- 对法规/检查：形成可追溯、可检索、可导出的 DHR 证据包。
- 对 IT/CSV：提供系统说明、权限矩阵、审计追踪说明、电子签名说明、测试脚本和发布记录，降低系统确认/验证负担。
- 对管理层：以低于重型 MES 的成本和周期，优先解决 DHR 与放行证据问题。

### 2.4 商用成功指标

| 指标 | 商用目标 | 测量方式 |
| --- | --- | --- |
| 试点上线周期 | 单产品族 4-8 周内完成首批 DHR 上线 | 项目里程碑 |
| QA 审核效率 | 试点产品 DHR 审核时间降低 30%-50% | 上线前后采样 |
| 放行前缺项发现率 | 已配置必填/签名/异常项 100% 被系统识别 | 完整性检查日志 |
| 归档包准备效率 | 已放行 DHR 5 分钟内生成归档包 | 系统日志 |
| 电子记录留痕覆盖率 | 受控操作 100% 审计追踪 | 测试与运行抽样 |
| 客户验证支持 | 每个正式交付版本均提供供应商验证支持包 | 发布检查清单 |
| 产品复用率 | 行业模板包复用率逐步提升，减少纯定制 | 实施统计 |

## 3. 产品定位与边界

### 3.1 产品定位

本产品是 eDHR 系统，不是完整 MES、QMS、ERP、WMS、LIMS、DMS 或 UDI 平台。它的核心对象是 DHR 证据链：

```text
受控产品/模板
-> DHR 实例
-> 生产记录
-> 检验记录
-> 物料/部件/序列追溯
-> 异常/返工/让步/紧急放行关联
-> QA 审核
-> 产品放行
-> 归档导出
```

### 3.2 对外推荐描述

> 基于 2025 版医疗器械 GMP 的专业 eDHR 系统，支持电子生产记录、检验记录、追溯、例外审核、产品放行、审计追踪、电子签名、CSV/CSA 支持和检查证据包导出。

### 3.3 明确不做

- 不承诺“使用本系统即可保证 GMP 合规”。
- 不自动替代质量负责人或授权放行人员判断。
- 不做完整生产排产、OEE、APS、复杂派工。
- 不做完整库存、采购、财务、成本核算。
- 不做完整 CAPA、投诉、供应商质量、变更控制和内审管理。
- 不做完整实验室 LIMS。
- 不做完整文控 DMS。
- 不做完整 UDI 申报平台。

### 3.4 和其他系统的关系

| 系统 | eDHR 关系 |
| --- | --- |
| ERP | 接收工单、产品、BOM、批号、计划数量；返回 DHR/放行状态 |
| MES | 可接收生产上下文；不替代 MES 的排产、派工和设备执行 |
| WMS | 校验物料批号、状态、效期；不替代库存事务 |
| QMS | 关联偏差、不合格、CAPA、变更编号；不替代 QMS 流程 |
| DMS | 引用 SOP、工艺规程、检验规程编号和版本；不替代文控生命周期 |
| LIMS | 接收或挂接检验结果、报告、证书；不替代样品和实验室全流程 |
| UDI 平台 | 记录和核对 UDI/PI；不替代监管数据库申报 |

## 4. 目标客户与用户角色

### 4.1 首选客户画像

- 国内二类、三类医疗器械生产企业。
- 纸质批记录或纸电混合记录较多。
- QA 放行审核慢、缺项多、跨部门取证成本高。
- 对新版 GMP、客户审计、注册人制度、委托生产或出海客户审计有压力。
- 暂时不想实施重型 MES，但需要快速提升 DHR 管理水平。

### 4.2 适合优先切入的细分场景

| 场景 | 适配原因 | 需要注意 |
| --- | --- | --- |
| IVD 试剂 | 批记录和检验记录多，放行压力明显 | 可能涉及环境、批量配制、留样和检验复杂度 |
| 一次性耗材 | 批次追溯、灭菌/包装/标签记录重要 | 无菌和灭菌证据需谨慎建模 |
| 医疗设备装配 | 序列号、关键部件、调试/检验记录重要 | 需要支持批次 + SN 混合结构 |
| 植入/介入类 | 追溯和放行证据价值高 | 质量风险高，首个试点需谨慎 |
| 委托生产 | 注册人/受托方证据协同痛点强 | 权限、责任和放行边界复杂 |

### 4.3 用户角色

| 角色 | 核心任务 | 关键诉求 |
| --- | --- | --- |
| 生产操作员 | 按步骤执行并填写记录 | 简单、快速、少打字、少返工 |
| 生产主管 | 查看进度、处理退回、监督缺项 | 实时状态、异常提醒 |
| 工艺工程师 | 配置工序、模板、规则 | 版本受控、可复用、少定制 |
| 检验员 | 填写/导入检验结果 | 项目清楚、设备/结果可追溯 |
| QA 审核员 | 审核 DHR 和异常 | 例外审核、证据可钻取 |
| 放行审核人 | 作出放行决定 | 放行条件清楚、责任边界清楚 |
| 质量负责人 | 管理质量证据和检查准备 | 审计追踪、签名、归档、检索 |
| IT/CSV 负责人 | 部署、运维、验证 | 供应商验证包、权限矩阵、安全说明 |
| 委托方 QA | 查看受托方生产放行证据 | 权限隔离、上市放行证据完整 |
| 受托方 QA | 完成生产放行并共享证据 | 责任边界清楚、证据传递高效 |

建议采用：

- 软件许可或订阅费：按站点、用户、模块、DHR 量或组合计费。
- 实施费：模板配置、主数据导入、权限矩阵、放行清单、试点上线。
- 年维护费：升级、问题修复、支持、验证支持包更新。
- 可选服务费：CSV/CSA 辅助执行、接口开发、行业模板包、客户审计支持。

避免：

- 一开始以纯项目外包方式销售。
- 承诺“所有表单都能无限定制且不加价”。
- 把 QMS/MES/WMS/LIMS 的工作免费并入 eDHR 项目。

## 7. 核心业务流程

### 7.1 模板受控流程

```text
创建模板草稿
-> 配置字段、规则、签名、复核要求
-> QA/工艺审核
-> 批准生效
-> 被 DHR 记录包引用
-> 变更时创建新版本
-> 旧版本作废但历史 DHR 保持可读
```

关键规则：

- 生效模板不可直接修改。
- 记录实例必须保存来源模板版本。
- 模板变更不自动改变已生成 DHR，除非走受控变更/补充流程。

### 7.2 DHR 执行流程

```text
接收或创建工单/批次/SN
-> 生成 DHR 实例
-> 分配生产/检验/复核任务
-> 操作员执行记录
-> 系统校验必填、范围、扫码、签名
-> 复核人复核
-> 异常项进入处理或挂接
-> DHR 提交 QA 审核
```

### 7.3 QA 审核与放行流程

```text
DHR 提交审核
-> 系统执行完整性检查
-> QA 查看例外项
-> 退回或关闭问题
-> 放行审核人查看放行清单
-> 签署放行决定
-> DHR 冻结
-> 生成归档包
```

### 7.4 委托生产流程

```text
受托方执行生产记录
-> 受托方生产放行
-> 委托方查看生产和质量控制证据
-> 委托方完成上市放行
-> 双方保留相关审计追踪和放行证据
```

该流程属于 v2.0 或特定客户增强能力，v1.0 只需预留角色、数据边界和放行模型。

## 12. CSV/CSA 与客户/供应商责任

### 12.1 客户责任

- 明确系统预期用途。
- 批准用户权限和职责。
- 确认模板、规则、放行条件和记录保存策略。
- 执行或批准系统确认/验证。
- 维护 SOP、培训、变更控制和周期性复核。
- 对最终产品放行和质量判断负责。

### 12.2 供应商责任

- 提供符合产品声明的功能。
- 提供系统说明、架构、数据流、权限矩阵、审计追踪说明、电子签名说明。
- 提供发布记录、变更影响说明、测试脚本模板和验证支持资料。
- 提供备份/恢复和部署建议。
- 明确产品限制和已知问题。

### 12.3 不应承诺

- 不承诺代替客户完成全部验证。
- 不承诺系统自动证明 GMP 合规。
- 不承诺系统自动判断产品可放行。
- 不承诺供应商承担客户质量体系责任。

## 14. 集成策略

### 14.1 总原则

- 商用版必须能独立运行。
- 集成增强效率，但不应成为首个客户上线的必要前提。
- 所有接口导入数据都必须校验、映射、留痕。
- 外部系统数据进入 DHR 后，应形成受控证据或受控引用。

### 14.2 集成优先级

| 优先级 | 集成 | 原因 |
| --- | --- | --- |
| P0 | CSV/Excel 导入、基础 API | 支持快速试点和私有化交付 |
| P1 | ERP/MES 工单和产品数据 | 减少重复录入 |
| P1 | QMS 事件编号和状态 | 放行审核需要异常闭环信息 |
| P1 | DMS 文件编号和版本 | 模板和记录需要引用受控文件 |
| P1 | WMS/ERP 物料状态 | 物料放行状态和效期校验 |
| P1 | LIMS 检验结果 | 检验数据进入 DHR 证据链 |
| P2 | 设备/仪器采集 | 提高自动化，但实施差异大 |

## 16. AI 路线边界

AI 不属于 v1.0 商用必需能力。

未来可选能力：

- OCR 辅助历史纸质记录迁移。
- AI 辅助检查 DHR 缺项和证据分类。
- AI 辅助生成验证测试草稿或配置建议。

严格限制：

- AI 不得静默修改受控记录。
- AI 不得作出最终放行决定。
- AI 建议必须有来源、版本、置信度或不确定性提示。
- AI 输出必须人工确认后才能进入受控记录。
- 客户数据是否进入模型训练必须由合同明确约定，默认不得用于训练。

## 18. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 商用范围过宽变成轻 MES | 交付变慢、竞争变复杂 | 保持 eDHR 证据链优先，MES 能力只做必要上下文 |
| 客户要求无限表单定制 | 产品化失败 | 建立模板能力边界和行业模板包策略 |
| 电子签名方案被客户/审计质疑 | 上线受阻 | 早期完成法律/质量/IT 联合评审，提供签名说明和 SOP 建议 |
| 审计追踪不可变性不足 | 合规风险 | 审计事件独立模型、无普通删除入口、导出可核验 |
| 私有化部署支持成本高 | 毛利下降 | 标准部署包、升级包、监控和备份流程 |
| 接口项目化过重 | 实施成本失控 | 先标准 API/导入，再做客户接口适配 |
| 委托生产权限设计不严 | 数据泄露和责任混乱 | v1.0 预留模型，v2.0 专项设计 |
| 产品宣称过度 | 法规和销售风险 | 使用受控宣称，不说保证合规或自动放行 |

## 19. PRD 自检

### 产品/创始人视角

正式商用版边界清晰：以 eDHR 证据链、审核放行和 GMP 2025 电子记录要求为主，不把 MES/QMS/WMS/LIMS 吞进来。商业包装需要在“标准产品 + 模板包 + 有边界实施”上保持纪律。

### 最终用户视角

生产操作员需要低摩擦输入，QA 需要例外审核和证据钻取，质量负责人需要审计追踪、签名、归档和检查包。商用版如果只强调合规而忽略现场体验，会影响落地。

### 质量/法规视角

PRD 将新版 GMP 2025 的电子记录、软件确认、生产记录、检验记录、追溯、放行和委托生产作为核心映射，并避免保证合规、自动放行等过度宣称。

### 工程视角

v1.0 应先保证核心闭环和底座稳定，接口、行业包、委托生产和设备采集按路线图推进。最关键工程风险是模板引擎、审计追踪、电子签名、状态机和归档包的模型设计。

### 数据安全/运维视角

私有化部署优先符合国内医疗器械客户信任预期，但需要标准部署、备份、恢复、升级和日志方案，否则交付成本会很快上升。

### 评分

当前 PRD 成熟度估算：87 / 100。

剩余 P1 风险：

- 首个行业模板包尚未确定。
- 电子签名法律技术方案需要专项设计。
- 私有化部署架构、安全基线和运维 SLA 需要技术规格。
- CSV/CSA 文档包需要单独落地为可交付模板。

## 20. 待创始团队确认事项

1. 第一商用版本命名与包装：Foundation/Professional/Enterprise 是否合适。
2. 第一个行业模板包选择：IVD、耗材、设备装配、无菌/灭菌，还是委托生产。
3. 私有化部署的技术栈、数据库、中间件和运维支持边界。
4. 电子签名最低实现和未来增强路径。
5. 是否在 v1.0 就支持批次 + 序列号混合 DHR。
6. 第一个客户必须集成的系统清单。
7. CSV/CSA 是“仅提供文档模板”还是“提供辅助验证服务”。
8. 行业模板包是否作为单独商业模块收费。

## 21. 剩余假设

- 首批客户主要在中国大陆市场。
- 客户真实生产质量记录更倾向私有化或专有云部署。
- 第一阶段先做 eDHR 合规基座和核心闭环，再让业务专家补充细分场景模板。
- 产品自身作为 GxP 相关信息化系统，需要被客户进行确认/验证。
- 本产品供应商提供支持工具和证据，不承担客户最终质量判断和法规责任。

## 22. 非法律意见说明

本文件为医疗器械 eDHR 商用产品需求草案，不构成法律、法规、质量体系、电子签名合法性或检查通过意见。对外销售、客户上线、系统确认/验证、电子签名实现和 GMP 条款适用性，应由具备医疗器械法规、质量体系、计算机化系统验证、数据安全和法务经验的专业人员评审。
