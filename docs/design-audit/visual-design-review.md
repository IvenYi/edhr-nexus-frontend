# eDHR 前端视觉设计全面审查报告

> **审查日期**: 2026-06-06
> **审查范围**: 前端全局样式、布局框架、Dashboard、登录页、列表/表格页面、流程中心、系统管理
> **技术栈**: React 18 + MUI v5 + Tailwind CSS v3 + TypeScript + Vite
> **产品定位**: 商业级 eDHR（电子设备历史记录）— 面向医疗器械/GMP 合规场景的 B2B 生产管理平台

---

## 一、总体评价

当前前端在**工程架构**层面是合格的——组件拆分清晰、路由懒加载、React Query 状态管理规范。但在**视觉设计**层面，整体停留在 MUI 默认样式 + 少量手动调整的阶段，缺乏针对工业/GMP 软件的专业设计语言。

**核心问题一句话总结**：这是一套「能用的」界面，但不是一套「有品牌感的、让工程师信赖的」工业软件界面。

## 二、配色方案诊断与改进方案

### 2.1 现状问题

| 问题 | 现状 | 严重程度 |
|------|------|----------|
| Error 色用了橙色 | `#E65100` (Deep Orange) | 🔴 严重 |
| 主色偏暗偏灰 | `#0D6EAA` 缺乏工业软件的「精密感」 | 🟡 中等 |
| 背景色缺乏层次 | `#EFF2F7` 与功能菜单 `#F7F9FC` 几乎无法区分 | 🟡 中等 |
| 深色侧边栏割裂感强 | `#1B2A4A` 深海军蓝与浅色主体反差过大 | 🟡 中等 |
| 成功色不统一 | 快捷入口用 `#388e3c`，主题 secondary 用 `#00897B` | 🟠 轻微 |
| 硬编码颜色散落各处 | 代码中大量 `'#0D6EAA'`、`'#E65100'` 等字面量 | 🟠 轻微 |

### 2.2 改进方案：工业级配色系统

#### 2.2.1 推荐色板

```
┌─────────────────────────────────────────────────────────┐
│  Primary (品牌蓝)    #1565C0 → #0D47A1                   │
│  Secondary (辅助绿)  #00897B → #00695C                   │
│  Success (成功/通过)  #2E7D32                             │
│  Warning (警告/待处理) #F57F17                            │
│  Error (错误/退回)   #C62828  ← 必须改为红色               │
│  Info (信息/运行中)  #0277BD                              │
│                                                         │
│  Surface (页面背景)   #F0F2F5                             │
│  Surface Card (卡片) #FFFFFF                              │
│  Surface Elevated    #FAFBFC                              │
│                                                         │
│  Sidebar Dark        #121C2D (更深、更专业)                │
│  Sidebar Accent      #1A2942                              │
│                                                         │
│  Text Primary        #1A2332                              │
│  Text Secondary      #5A6878                              │
│  Text Disabled       #8E9BAF                              │
│  Divider             #E2E6EC                              │
└─────────────────────────────────────────────────────────┘
```

#### 2.2.2 关键修改

**① Error 色从橙色改为红色**
- 现状：`error: { main: '#E65100' }`（深橙色）
- 问题：GMP 系统中，「错误」「退回」「作废」都应该是红色语义。橙色在世界范围内代表「警告/注意」
- 方案：`error: { main: '#C62828' }`（Material Red 800）

**② Primary 色加深并统一**
- 现状：`#0D6EAA` 偏灰，不够「工业精密」
- 方案：`#1565C0`（Material Blue 800），更浓郁、更有信任感

**③ 背景色增加层次**
- 现状：页面背景 `#EFF2F7`，功能菜单 `#F7F9FC`，区分度极低
- 方案：
  - 页面背景稍微偏灰 `#F0F2F5`
  - 功能菜单背景加一条细微右边框阴影，或使用 `#F5F7FA` + 1px 灰度加深
  - 卡片增加微弱阴影提升层级

**④ 侧边栏改用更深的专业色**
- 现状：`#1B2A4A`
- 方案：`#121C2D`（参考 Linear/Vercel 风格），激活态用半透明 primary 色覆盖

## 三、字体排版诊断与改进方案

### 3.1 现状问题

| 问题 | 现状 | 严重程度 |
|------|------|----------|
| 基础字号偏小 | `fontSize: 14` | 🟡 中等 |
| 页面标题层级弱 | `variant="h5"` 在数据密集页面中显得不够突出 | 🟡 中等 |
| 中文字体优先级不够 | `Inter → Roboto → Noto Sans SC`，中文本应前置 | 🟠 轻微 |
| 缺乏字重对比 | h6 仅设置了 `fontWeight: 600`，与正文区分度不足 | 🟠 轻微 |
| 表格内文字密度无控制 | 无字号、行高定制 | 🟠 轻微 |

### 3.2 改进方案

```typescript
typography: {
  fontFamily: "'Noto Sans SC', 'Inter', 'Roboto', -apple-system, sans-serif",
  fontSize: 14.5, // 微妙增大
  h4: { fontSize: 28, fontWeight: 700, letterSpacing: '-0.02em' },  // 页面主标题
  h5: { fontSize: 22, fontWeight: 600, letterSpacing: '-0.01em' },  // 区块标题
  h6: { fontSize: 17, fontWeight: 600 },                             // 卡片标题
  body1: { fontSize: 14.5, lineHeight: 1.6 },
  body2: { fontSize: 13, lineHeight: 1.5 },
  caption: { fontSize: 12, lineHeight: 1.4 },
}
```

关键改动：
- **中文字体前置**：`Noto Sans SC` 放在第一位，加载更快、中文渲染更好
- **增大基础字号**：14 → 14.5px，对数据密集型工业软件更友好
- **增加字母间距**：h4/h5 负间距让标题更紧凑有力
- **Line-height 显式设定**：避免 MUI 默认行高在不同场景下不一致

## 四、布局与信息架构优化

### 4.1 现状问题

| 问题 | 分析 | 严重程度 |
|------|------|----------|
| 三栏布局的实际面积利用低 | 模块栏 72px + 功能菜单 220px + 内容区，但功能菜单仅用了一半空间 | 🟡 中等 |
| 缺少面包屑导航 | 深层页面（如流程实例详情）用户不知道自己在哪里 | 🟡 中等 |
| 页面标题样式不统一 | 有的用 `h5`，有的用 `h5 + mb`，有的没有 | 🟠 轻微 |
| 内容区域没有最大宽度限制 | 在大屏幕上表格拉伸到无边无际 | 🟠 轻微 |
| Header 高度偏矮 | 56px 对企业级应用偏紧凑 | 🟠 轻微 |

### 4.2 改进方案

#### 4.2.1 增加页面标题栏组件（PageHeader）

每个页面顶部统一使用 PageHeader：

```
┌──────────────────────────────────────────────────────┐
│ 面包屑: 首页 / 系统管理 / 用户管理                      │
│                                                      │
│ 用户管理                    [+ 新增用户]  [导入] [导出] │
│ 管理系统用户、角色与权限配置                            │
└──────────────────────────────────────────────────────┘
```

#### 4.2.2 表格页面标准化布局

```
┌──────────────────────────────────────────────────────┐
│ PageHeader (面包屑 + 标题 + 操作按钮)                   │
├──────────────────────────────────────────────────────┤
│ FilterBar (搜索框 + 筛选下拉 + 状态筛选)                │
├──────────────────────────────────────────────────────┤
│ Table (带斑马纹、hover 态、固定表头)                     │
├──────────────────────────────────────────────────────┤
│ TableFooter (共 N 条 | 每页 M 条 | 分页器)              │
└──────────────────────────────────────────────────────┘
```

#### 4.2.3 内容区域宽度限制

对企业级 SaaS，推荐在 1440px+ 屏幕上设定 `max-width: 1280px` 并居中，避免表格列无限拉伸。

### 6.1 全局问题

| 问题 | 严重程度 | 改进方案 |
|------|----------|----------|
| `alert()` 弹窗 | 🔴 严重 | 统一使用 Snackbar / Toast 组件 |

### 6.2 关键交互流改进

**任务处理流程 (TaskDetail)**：
- 现状：审核意见 + 密码 + 四个按钮平铺
- 改进：底部固定操作栏（sticky footer），区分主操作（通过/退回）和次操作（转办/提交），增加操作二次确认

**表单弹窗**：
- 现状：编辑/新增共用同一 Dialog，打开无过渡动画
- 改进：Dialog 增加 enter 过渡，表单校验前移（实时校验），保存按钮在表单有效前 disabled

## 七、代码层面改进建议

### 7.1 颜色 Token 统一

当前散落各处的硬编码颜色应集中到 MUI Theme：

```typescript
// 全部颜色都应通过 theme.palette 或自定义变量引用
// ❌ 错误
sx={{ color: '#0D6EAA' }}
sx={{ bgcolor: '#E65100' }}

// ✅ 正确
sx={{ color: 'primary.main' }}
sx={{ bgcolor: 'error.main' }}
```

### 7.2 Tailwind CSS 的去留

现状：同时安装了 Tailwind 和 MUI，但 Tailwind 几乎没有使用（仅 index.css 中 `@tailwind` 指令）。`preflight: false` 让 Tailwind 的 base reset 也被禁用。

**建议**：
- **短期**：继续以 MUI `sx` 为主要样式手段，Tailwind 仅用于少数需要快速原型的场景
- **长期评估**：如果团队倾向 Tailwind-first，考虑迁移到 Tailwind + Headless UI 或其他无样式组件库，避免 MUI 和 Tailwind 的样式冲突

### 7.3 抽取公共模式

以下模式在多个页面中重复出现，建议抽取：

| 模式 | 出现次数 | 建议 |
|------|----------|------|
| 页面标题 + 操作按钮 | 10+ | `PageHeader` 组件 |
| CRUD 表格 + Dialog | 8+ | `CrudPage` 或 `useCrudTable` hook |
| 筛选栏 | 2 | `FilterBar` 组件 |
| 空状态 | 5+ | `EmptyState` 组件 |
| 加载骨架 | 0 | `TableSkeleton` 组件 |

## 九、总结

当前前端在**功能完整度**上没有大问题——CRUD、流程、权限的核心交互都已实现。但在**视觉专业度**上距离商业化 eDHR 产品还有差距。

核心改进方向：
- **颜色系统**：从「MUI 默认」升级为「工业品牌色」
- **字体排版**：从小而密升级为清晰有层级
- **组件一致性**：抽取公共模式，统一交互行为
- **细节打磨**：骨架屏、空状态、防抖、二次确认

每一个改进点都是可独立实施的小任务，可以按优先级逐步迭代，不阻塞功能开发。
